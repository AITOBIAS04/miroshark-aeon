### 1. Simulation Fork ("Explore a Variant")

**Type:** Feature
**Effort:** Small (hours)
**Impact:** When a simulation produces a surprising consensus — "72% Bullish on a scenario most would have called neutral" — the natural next question is: *was that robust?* What if I'd used 40 agents instead of 20? 15 rounds instead of 10? A different model? Right now answering that requires manually re-entering the full scenario config. Fork collapses that friction to one click: a "Fork & vary" button on any published sim's watch page pre-fills the new-simulation form with the parent's full config (scenario text, agent count, rounds, model, archetypes) while leaving all fields editable. The forked sim records `parent_sim_id` in its `config.json` and shows a "Forked from [original]" chip on the watch page that links back. The parent watch page gains a "Forks (N)" chip that lists child sims. For researchers building sensitivity tables — "we ran the same scenario at 10, 20, and 40 agents; here are the three consensus outcomes" — this is the reproducibility primitive the platform currently lacks. For operators, it's a one-click "run this again with a twist."

**How:**
1. Add `parent_sim_id: str|null` to the simulation launch schema (`POST /api/simulation/start`) and to `<sim_dir>/config.json`. When `parent_sim_id` is provided at launch, validate that it references a published simulation (return HTTP 400 if not found or not published). Write `parent_sim_id` to the new sim's config. Add `GET /api/simulation/:id/forks` returning `{parent_sim_id: str|null, forks: [{sim_id, published_at, consensus_direction, confidence_score, n_agents, n_rounds}]}` — list of all sims that reference this sim as `parent_sim_id`, sorted by `published_at` descending, with confidence_score pulled from `confidence.json` if present. In `simulation.py`, add a `forks_count: int` field to the detail endpoint and gallery card payload (count of published sims with matching `parent_sim_id` — scan efficiently by maintaining a reverse index in `<data_dir>/fork_index.json`: `{parent_sim_id: [child_sim_id, ...]}` updated atomically on each launch).
2. In `Step3Simulation.vue`, add a "Fork & vary" button in the share/action toolbar (alongside Regenerate, Publish, Download). The button navigates to `/new?fork=<sim_id>`. In `Step1GraphBuild.vue` (the new-simulation form), on mount detect `?fork=<sim_id>`: call `GET /api/simulation/:id` to fetch the parent's config, then pre-fill all form fields (scenario textarea, agent count, rounds, model selector, archetype multiselect). Show a dismissable banner: "Forking from [scenario excerpt…] — edit any field before running." In `SimulationWatchPage.vue` (or equivalent), add a "Forked from [→ link]" chip in the sim header when `parent_sim_id` is set. Add a "Forks (N)" chip that opens a drawer listing the child sims from `GET /api/simulation/:id/forks` — each entry shows consensus direction chip, confidence badge, agent count, and a "View →" link.
3. Add `GET /api/simulation/:id/forks` to the OpenAPI spec under the "Simulation" tag with a `ForksResponse` schema: `{parent_sim_id: string|null, forks: [{sim_id: string, published_at: string, consensus_direction: string, confidence_score: integer|null, n_agents: integer, n_rounds: integer}]}`. Add `parent_sim_id: string|null` and `forks_count: integer` to the simulation detail response schema. Add 12 unit tests in `backend/tests/test_unit_fork.py`: fork with valid parent writes `parent_sim_id`, fork with missing parent returns 400, fork with unpublished parent returns 400, `GET /api/simulation/:id/forks` returns child list, forks_count in detail response, fork_index written on first fork, fork_index appended on second fork, fork_index creation idempotent (same child not duplicated), confidence_score null when no confidence.json, sorted by published_at descending, parent_sim_id null when no fork, route decorator presence.

### 2. Outbound Completion Webhook

**Type:** Integration
**Effort:** Small (hours)
**Impact:** The inbound launch webhook (already built) lets external systems trigger simulations via HTTP. But those systems still have to poll `GET /api/simulation/:id` repeatedly to know when the sim is done — a pattern that burns API calls and adds latency. An outbound completion webhook inverts this: at launch, operators optionally provide a `completion_webhook_url` (HTTPS only), and when the simulation finishes MiroShark fires a POST with the full result payload. Zero polling. A developer who triggers a sim at the end of their analysis pipeline gets the result pushed to their system automatically — the sim result becomes a trigger for the next step. For operators using MiroShark as a backend service (rather than a web UI), this is the integration pattern they'd expect: fire and receive, not fire and poll. Pairs naturally with the Inbound Launch Webhook: inbound webhook launches the sim, completion webhook delivers the result — MiroShark becomes a passthrough node in a larger automation chain.

**How:**
1. Add `completion_webhook_url: str|null` to the simulation launch schema (`POST /api/simulation/start`) and to `<sim_dir>/config.json`. Validate at launch time: must start with `https://` (reject `http://` — security requirement), must be a valid URL per `urllib.parse.urlparse` (scheme, netloc both non-empty), must not be an internal address (reject `localhost`, `127.0.0.1`, `0.0.0.0`, any `192.168.x.x` or `10.x.x.x` — prevents SSRF). Store in config; write a `.pending-completion-webhook/<sim_id>.json` file on simulation completion: `{sim_id, completion_webhook_url, payload: {sim_id, status: "complete", consensus_direction, yes_pct, no_pct, neutral_pct, confidence_score, quality_health, share_url, completed_at: ISO}}`. Add `scripts/postprocess-completion-webhook.sh`: reads each `.pending-completion-webhook/*.json`, fires `curl -X POST -H "Content-Type: application/json" -d @<payload_file> <completion_webhook_url>` with a 10-second timeout and max 3 retry attempts (exponential backoff: 2s, 4s, 8s). On success, moves the file to `.pending-completion-webhook/sent/`. On final failure, moves to `.pending-completion-webhook/failed/` and logs the HTTP status code.
2. Add `completion_webhook_url` to the launch form in `Step1GraphBuild.vue` as an optional "Advanced" section (collapsed by default): a text input with placeholder `https://your-endpoint.com/webhook`, a brief description: "We'll POST the simulation results here when it finishes — HTTPS only." Show a validation error inline if the URL doesn't start with `https://`. In `EmbedDialog.vue` (or wherever the integration chip section lives), add a "Completion Webhook" chip alongside the Trading Signal and MCP chips: a short description ("Get results pushed to your endpoint on completion — set at launch time") with a link to `docs/INTEGRATIONS.md#completion-webhook`. Add a note in `Step3Simulation.vue` results panel when `completion_webhook_url` is set: a small chip "Completion webhook configured — results will be POSTed to your endpoint."
3. Add `completion_webhook_url: string|null` to the OpenAPI spec for `POST /api/simulation/start` request schema, with the description: "Optional HTTPS URL to receive a POST with simulation results on completion. Must be HTTPS. Set once at launch — cannot be changed after simulation starts." Document the completion payload schema as `WebhookCompletionPayload` in the OpenAPI spec. Add 12 unit tests in `backend/tests/test_unit_completion_webhook.py`: HTTPS URL accepted, HTTP URL rejected, localhost rejected, 127.0.0.1 rejected, 192.168.x.x rejected, malformed URL rejected, valid URL stored in config.json, pending file written on completion, payload shape (all required fields present), retry logic (mock curl exits 1 on first 2 calls, succeeds on 3rd), final failure moves to /failed/, route decorator presence. In `docs/INTEGRATIONS.md` (create if absent), add a "Completion Webhook" section: payload schema, retry behavior, HTTPS requirement, SSRF protection note.

### 3. AI Narrative Summary ("Explain This Simulation")

**Type:** Feature
**Effort:** Small (hours)
**Impact:** The simulation data layer is now deep — trajectory, sparklines, coalitions, confidence score, agent journey, interaction graph. What's missing is a plain-language bridge between those signals and a non-expert reader. A governance researcher who runs a 20-agent simulation on a Senate vote sees 72% YES, a coalition of 6 progressives, and a Risk Manager who flipped in round 4 — but synthesizing what that means requires reading the agent posts and charting the trajectory by hand. The "Explain This Simulation" button calls Claude with the full structured sim output and returns a 150–200 word narrative: what happened, which agents drove the outcome, what the key pivot moment was, and what confidence level to attach to the result. The output is written to `<sim_dir>/narrative.md` (cached — never regenerated unless manually cleared) and shown in a collapsible panel on the watch page. For operators, it's a first-draft interpretation they can edit and publish. For the academic citation trajectory, a human-readable analysis paragraph that accompanies the data makes the simulation result accessible to non-quant reviewers. MiroShark already calls Claude to simulate agents — this is one more call at the end, using the same infrastructure.

**How:**
1. Add `backend/app/services/narrative_service.py` (~150 LoC). The function `generate_narrative(sim_id, data_dir, api_key)` reads `trajectory.json` (final consensus: yes_pct, direction, round-by-round arc), `coalitions.json` (coalition count, sizes, labels if present), `confidence.json` (total confidence_score, component breakdown), `adversarial_report.json` if present (robustness tier), and the top 5 agent posts by influence count from `actions.jsonl`. Composes a structured prompt to the Anthropic API (Claude Haiku — cheapest option, sufficient for summarization): "You are analyzing a MiroShark swarm simulation. Here is the data: [structured JSON]. Write a 150-200 word plain-language narrative explaining: (1) the final consensus and what it means, (2) which agents or coalitions were most influential, (3) the key pivot moment (the round with the largest single-round YES% shift), (4) a one-sentence confidence assessment based on the confidence score. Write for an intelligent non-expert reader. Do not use jargon. Do not mention that you are an AI." Uses `anthropic.Anthropic(api_key=api_key)` with `ANTHROPIC_API_KEY` from env. Writes the response to `<sim_dir>/narrative.md`. Returns the narrative string. Add `GET /api/simulation/:id/narrative` (admin-token gated for generation, public for reading): if `narrative.md` exists, return `{sim_id, narrative: str, generated_at: ISO (from file mtime)}` immediately (cached). If not, call `generate_narrative` (may take 2–4s) and return. Add `POST /api/simulation/:id/narrative/regenerate` (admin-token gated) that deletes `narrative.md` and calls `generate_narrative` fresh — for when operators want to update the narrative after editing the simulation or changing context.
2. In `Step3Simulation.vue`, add a "Narrative Summary" collapsible panel in the results layout — between the Confidence panel and the Share/Embed section. Panel header: "Narrative Summary" with an "✨ Explain" button (shown when no narrative exists, admin-gated) and a "↺ Regenerate" button (shown when narrative exists, admin-gated). Loading state: animated "Analyzing simulation..." text. On load, fetch `GET /api/simulation/:id/narrative`; if it returns a narrative, render it as a styled prose block (not a table, not a chart — just text, `font-serif text-base leading-relaxed max-w-prose`). Below the narrative, a small "Generated by Claude Haiku" attribution line in `text-xs text-gray-400`. Non-admin visitors (no admin token) see the narrative if it exists, but not the "Explain" or "Regenerate" buttons — the narrative is public once generated, generation is gated. In `EmbedDialog.vue`, add a "Narrative" chip in the data exports section: "AI-generated plain-language summary — visible on the watch page and included in Archive Bundle."
3. Add `GET /api/simulation/:id/narrative` and `POST /api/simulation/:id/narrative/regenerate` to the OpenAPI spec under the "Analytics" tag. Add `has_narrative: bool` to the gallery card payload (from checking existence of `narrative.md`). Include `narrative.md` in the Archive Bundle manifest (alongside confidence.json, coalitions.json, and the other derived files). Add 10 unit tests in `backend/tests/test_unit_narrative.py`: narrative cached on second call (generate not called twice), missing trajectory.json returns graceful error, narrative written to correct path, `has_narrative` false when file absent, `has_narrative` true when file present, regenerate deletes old file and triggers new generate, public GET returns narrative without admin token, GET returns 404 when no narrative and no admin token (no auto-generate for public requests — generation only on explicit admin request), admin token triggers generate, route decorator presence. In `docs/FEATURES.md`, add a "AI Narrative Summary" section: explains the prompt pattern, the Claude Haiku model selection and why, the caching behavior, and the "public read / admin generate" access model.

### 4. Operator Dashboard ("All My Simulations")

**Type:** DX improvement
**Effort:** Small (hours)
**Impact:** An operator who has run 30 simulations over the past month has no view of their own history in MiroShark. They navigate by bookmarks, by the gallery (which only shows published sims), or by remembering sim IDs. The public gallery sorts by recency across all operators — finding "the 8-agent sim I ran two weeks ago on the ECB decision" requires scrolling or free-text search. The Operator Dashboard fixes this: an admin-gated `/my-simulations` page that lists every simulation created with the current admin token (keyed by a non-reversible hash of the token so the raw token is never stored). Columns: status (running/published/private), scenario excerpt, agent count, confidence score, embed hits, last modified. Sortable, filterable by status. A single-view command center for operators who use MiroShark regularly. This is the "my account" equivalent for a token-based auth system — no OAuth required, no user accounts, just the existing admin token as identity. For power operators running recurring series and curator collections, it replaces bookmark-based navigation with a proper inventory.

**How:**
1. Add `backend/app/services/operator_dashboard_service.py` (~200 LoC, pure stdlib). The service takes `token_hash = hashlib.sha256(admin_token.encode()).hexdigest()[:16]` as the operator identifier (never stored raw). At launch (`POST /api/simulation/start`), when an admin token is validated, append `token_hash` to `<sim_dir>/config.json`. Add `GET /api/operator/simulations` (admin-token gated): reads the admin token from the request, computes `token_hash`, scans all sim dirs for `config.json` entries where `token_hash` matches (using the same scan loop as the gallery endpoint but without the published filter), and returns `{sims: [{sim_id, status, scenario_excerpt (first 120 chars), n_agents, n_rounds, model, confidence_score: int|null, embed_hits: int (from surface_stats.json), published_at: ISO|null, created_at: ISO, last_modified: ISO (from dir mtime)}], total: int, published_count: int, private_count: int, running_count: int}` sorted by `last_modified` descending. Cache the scan result in-memory per request (no file cache — the result is token-specific and changes frequently). Add `GET /api/operator/stats` (admin-token gated): aggregate view — total sims, avg confidence score, total embed hits across all operator sims, most-embedded sim (`{sim_id, scenario_excerpt, embed_hits}`), highest-confidence sim.
2. Add `OperatorDashboardView.vue` as a new page at `/my-simulations` (admin-gated: redirect to `/` if no admin token in localStorage). Page header: "My Simulations" with a summary stat row (total: N, published: N, private: N, running: N). Below: a status filter tab row (All / Published / Private / Running). Main content: a sortable table (default sort: last modified desc) with columns — Status chip (Published/Private/Running), Scenario (truncated 120 chars, click to navigate to `/watch/<id>` or `/sim/<id>` if running), Agents, Rounds, Confidence (numeric badge, gray if null), Embed Hits, Last Modified (relative: "3 days ago"). Clicking any row navigates to the simulation. In the page sidebar or footer, show the aggregate stats from `GET /api/operator/stats`: total embed hits, avg confidence score with a color band, and the "top sim" card. Add a `/my-simulations` link in the app navigation (only visible when admin token is set in localStorage — same pattern as admin-only buttons elsewhere).
3. Add `GET /api/operator/simulations` and `GET /api/operator/stats` to the OpenAPI spec under a new "Operator" tag with `OperatorDashboardResponse` and `OperatorStatsResponse` schemas. Add 12 unit tests in `backend/tests/test_unit_operator_dashboard.py`: token_hash derivation is deterministic (same token → same hash), token_hash is non-reversible (hash output does not contain the raw token substring), only sims with matching token_hash returned, sim with no token_hash excluded, status field derivation (running: status file present + no published_at, published: published_at set, private: neither), confidence_score null when confidence.json absent, embed_hits 0 when surface_stats.json absent, last_modified from dir mtime, published_count / private_count / running_count aggregation correct, `GET /api/operator/stats` most-embedded sim correctly identified, admin token required (returns 403 without), route decorator presence.

### 5. Round-Specific Replay Permalink ("Share This Moment")

**Type:** Feature
**Effort:** Small (hours)
**Impact:** Discussion about simulations — in Telegram groups, Substack comments, academic notes — almost always references a specific moment: "the Risk Manager's flip in round 7 is the key insight." The watch page and replay player today have no way to deep-link to a specific round. Every share URL starts the replay at round 1 regardless of what the sender wanted to highlight. A `?round=N` query param makes the replay player initialize at a specific round (applying all round-by-round transitions up to N), and an optional `?agent=<name>` highlights a specific agent on load. A "Share this moment" button in the replay toolbar generates this URL and copies it to clipboard. This closes a critical gap in the social sharing UX: instead of "here is the simulation, go to round 7," it becomes "here is round 7" — the reader lands exactly where the story is. It also pairs naturally with the Discussion Threads feature (May 22 idea #4, not yet built): a comment can link to the exact round it's discussing via a round-permalink. For the annotation layer (already built), operators can generate round-specific links in their annotation text. Zero backend changes — this is entirely a frontend URL routing addition.

**How:**
1. In `SimulationWatchPage.vue` (or the equivalent page component that hosts the replay player), read `?round=N` and `?agent=<name>` query params on mount using the Vue Router `useRoute()` composable. Pass `initialRound: parseInt(route.query.round) || 1` to the replay player component as a prop. The replay player (wherever `currentRound` state is initialized) checks this prop on mount and fast-forwards to the specified round by replaying the round-by-round transition logic without animation (a `for (let r = 1; r < initialRound; r++) { applyRound(r) }` loop at initialization). When `?agent=<name>` is present, after initialization apply a "highlight" state to the named agent: pulse the node in `InteractionNetwork.vue` (same highlight pattern as the existing coalition-click behavior) and scroll the agent's belief history card into view in the agent list panel. Preserve both params in the browser URL bar as the user navigates rounds (push state on each round advance: `router.replace({ query: { ...route.query, round: currentRound } })`). This means a user who shares their URL mid-replay always shares their current round position.
2. In the replay toolbar (the round navigation controls in `Step3Simulation.vue` or wherever the replay player controls live), add a "Share this moment" button — a share/link icon that copies `window.location.href` (which includes the current `?round=N`) to clipboard and shows a "Link copied!" tooltip for 2 seconds. Add a "Jump to round" input field in the toolbar: a small numeric input (`1`–`max_rounds`) with an "→" submit button that calls the same fast-forward logic and updates the URL. This gives users a direct way to navigate to a known round without clicking through the timeline. In the agent list panel (wherever per-agent cards are shown), add a small "🔗" icon next to each agent name that generates and copies a URL with both `?round=<currentRound>&agent=<name>`. When the page loads with `?agent=<name>`, highlight the matching agent card with a `ring-2 ring-orange-400` border and scroll it into view with `scrollIntoView({ behavior: 'smooth' })`.
3. Add the `round` and `agent` params to the OpenAPI spec as query parameters on the `GET /embed/<sim_id>` route (the embed iframe also supports round-specific initialization, using the same `?round=N` param — the embed iframe's script reads it from `window.location.search` directly). Update `EmbedDialog.vue` to show the round-permalink field: below the standard iframe snippet, add an "Embed from a specific round" section — a number input (1 to max_rounds) + auto-updating snippet showing `?round=N` appended to the embed src URL. Add an end-to-end test note in `docs/FEATURES.md` under a "Replay Permalink" section: "Add `?round=7` to any watch page URL to start the replay at round 7. Add `?agent=Risk+Manager` to highlight that agent on load. Use the 'Share this moment' button in the replay toolbar to copy the current-round URL automatically. Works in embeds too: append `?round=7` to the embed `src`." Add 8 unit tests in `backend/tests/test_unit_replay_permalink.py` (backend test covers the embed route param passthrough): embed route preserves `?round` param in response (does not strip it), embed route preserves `?agent` param, out-of-range round param clamped to max_rounds, non-numeric round param ignored (defaults to 1), embed route returns 404 for unpublished sim regardless of params, `?round` with `?agent` both present handled, route decorator presence, CORS headers present on embed response.

## Selection Rationale

Five angles distinct from all 10 ideas suggested May 22 and May 24, the 25 push-blocked builds, and the open Railway PR:

- **Simulation Fork** (#1) — The GraphML export (May 24), trajectory CSV, and archive bundle all preserve what happened in one simulation. Fork is the first feature that uses an existing sim as the *starting point* for a new one — closing the sensitivity-analysis loop researchers need to go from "interesting result" to "is this robust?" The parent/child tracking turns the gallery into a versioned experiment tree rather than a flat list of independent runs.

- **Outbound Completion Webhook** (#2) — The inbound launch webhook (already built, push-blocked) handles the trigger side. The completion webhook handles the result side. Together they make MiroShark a composable service in larger automation pipelines — a developer can wire "trigger sim → receive result" without a single line of polling code. Distinct from the Trading Signal JSON (a pull endpoint), the Telegram/Discord notifications (channel-based, no per-sim targeting), and the MCP server (assistant-native, not pipeline-native).

- **AI Narrative Summary** (#3) — Every prior analytics feature (confidence score, calibration, sparklines, peak-round analytics) adds a new signal. The narrative summary adds interpretation. It's the first feature that speaks in plain English about what the data means — accessible to readers who can't interpret a confidence score breakdown or a coalition modularity score. Because MiroShark already calls Claude to simulate agents, the infrastructure cost is near-zero. The narrative is cached so it's a one-time API call per sim.

- **Operator Dashboard** (#4) — A blind spot in the current UX: operators have no view of their own simulation history. The gallery is sorted by recency across all operators; the admin-gated views are per-simulation; there's no inventory. The operator dashboard is the "my account" page for a token-based system — it surfaces the full history, status breakdown, and aggregate stats that operators need when using MiroShark as a recurring production tool rather than a one-off research toy.

- **Round-Specific Replay Permalink** (#5) — The most-requested latent behavior in any replay-capable tool: "link me to this exact moment." Pure frontend URL routing with zero backend changes and zero new dependencies. The `?round=N&agent=name` pattern is standard in video platforms (YouTube `?t=120`) and documentation tools — applying it to the simulation replay makes the data shareable at the grain users actually discuss it. Pairs with discussion threads, annotations, and embeds without requiring any of them to be built first.
