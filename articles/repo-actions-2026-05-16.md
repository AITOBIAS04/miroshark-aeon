## Ecosystem Context

MiroShark crosses 1,164 stars today — up 7 from yesterday, steady organic climb. The product layer is now unusually deep for an open-source simulation tool: 23 consecutive zero-new-dependency PRs, a full share-surface stack (PNG card, GIF, tweet thread, embed iframe, spectator watch, RSS feed, sitemap, SVG chart pending), on-chain provenance via OriginTrail DKG, and a researcher-handoff sequence (trajectory CSV, per-agent CSV, quality report, annotations, Jupyter notebook) that covers most of what an academic reviewer would ask for.

What that depth reveals is a gap at the edges: not in features, but in *access patterns*. Simulations can be published publicly or kept private — but there's no middle ground. Weekly activity has no summary surface. The gallery is comprehensive but unstructured; there's no way to organize a curated argument from a set of simulations. And the scenario text field still has no analytical feedback before you commit to a run.

Today's five ideas address the access and entry layers: how simulations get shared, how new users orient to the scenario field, how activity gets summarized, and how simulations get organized into arguments. Each is distinct from the 25 ideas suggested across May 10, 12, and 14, and from the chart SVG open PR.

### 1. Private Share Links with Expiry

**Type:** Feature
**Effort:** Small (hours)
**Impact:** Publishing a simulation makes it publicly indexed in the gallery and discoverable via sitemap. Not publishing keeps it invisible — the operator can see it, no one else can. There's no middle state: a time-limited URL that lets a specific collaborator view the full results without making the simulation gallery-public. The gap matters for practitioners: a consultant who runs a scenario for a client, a researcher who wants peer review before publication, a team testing a scenario internally. Private Share Links fill this precisely. `POST /api/simulation/:id/share-links` generates a token-signed URL (`/share/<token>`) that bypasses the publish gate and renders the full simulation result for any bearer. Each link carries a configurable expiry (24 hours / 7 days / never) and can be revoked individually without affecting other links or the simulation's public status. The Share dialog gains a "Private link" tab alongside "Embed" and "Cite" — a one-click flow that copies the link and shows its expiry. For operators running MiroShark on behalf of clients or collaborators, this is the share primitive that makes unpublished simulations shareable without the gallery commitment.
**How:**
1. Add a `share_links` table (or JSON file `<sim_dir>/share_links.json` — no SQLite dependency): `{token: str (32-char hex, urandom), expires_at: ISO|null, created_at: ISO, revoked: bool}`. Add `POST /api/simulation/:id/share-links` accepting `{expires_in: "24h"|"7d"|"never"}` and returning `{token, url: "/share/<token>", expires_at}`. Add `DELETE /api/simulation/:id/share-links/:token` to revoke. Add `GET /share/<token>` as a Flask route: validates token (existence, expiry, not revoked), then renders the full simulation result page identically to the published view — same components, same API calls, but with the token passed as a query param that the backend recognizes as a share-link bearer (bypassing the `is_public` check on data endpoints). Log each `/share/<token>` access in `<sim_dir>/surface_stats.json` under a new `private_share` counter. Add `GET /api/simulation/:id/share-links` returning all active links for the simulation (for the management UI).
2. In the Share / Embed dialog (`EmbedDialog.vue`), add a "Private link" tab as the first tab (before "Embed code" and "Chart SVG"). The tab shows: a "Generate link" button with expiry dropdown (24h / 7d / No expiry); after generation, a read-only URL input with a "Copy" button; below it, a managed list of active links — each row: masked token (`...abc123`), expiry label ("Expires in 6 days" / "No expiry"), a "Revoke" button. A note below the list: "Private links bypass gallery publishing — the simulation is not indexed or listed publicly." On the simulation detail page, add a "Share privately" chip below the main action buttons (alongside "Publish to gallery" / "Copy embed" / "Cite"). For simulations already published, the "Private link" tab shows a note: "This simulation is already published publicly — private links are redundant but still functional." This lets operators maintain expiring links for specific collaborators even on public sims (e.g., to track who accessed preliminary results before the public launch).
3. In `GET /api/simulation/:id` (the detail endpoint), add a `share_link_count: int` field showing how many active (non-expired, non-revoked) share links exist. This lets the management UI badge the Share button: "2 active private links." Add `POST /api/simulation/:id/share-links` and `GET /api/simulation/:id/share-links` and `DELETE /api/simulation/:id/share-links/:token` to the OpenAPI spec under a new "Private Sharing" tag. Document the bypass mechanism in `docs/API.md`: "Private share links grant read access to any bearer — treat them as passwords. Revoke immediately if shared in error." Add a `?share_token=<token>` query parameter to all data endpoints (`/trajectory.csv`, `/agent-trajectory.csv`, `/coalitions`, `/quality-report`, `/annotations`) so that an authenticated private-link request can fetch the simulation's data without the simulation being published. The `/share/<token>` page passes this token to all frontend API calls via a context variable.

### 2. Adversarial Stress-Test Mode

**Type:** Feature
**Effort:** Small (hours)
**Impact:** MiroShark's agents form their own consensus through peer influence and information exposure. The quality guard detects degenerate outcomes (lock, stagnation, neutral collapse, dominance), but it doesn't ask the harder question: *would this consensus have held if challenged?* A "74% YES" result is a very different claim depending on whether it emerged from a diverse, contested debate or from an echo chamber where no agent ever pushed back. Adversarial Stress-Test Mode answers this by designating 1–3 agents as adversarial at simulation launch: they're prompted to take the most rhetorically forceful contrarian position available against the emerging consensus, regardless of their initial archetype or platform. The simulation runs identically — same mechanics, same rounds, same quality guard — but the final report includes a **consensus robustness score**: what fraction of the run's consensus was maintained under direct adversarial challenge? "HIGH: consensus held at 71% YES despite 2 adversarial agents through round 12." "LOW: adversarial challenge caused a consensus reversal at round 8 — initial 60% YES collapsed to 38%." For researchers testing whether a scenario's consensus finding is robust or fragile, and for practitioners who want to stress-test a conclusion before presenting it, this is the experiment that can't be run with the current agent pool.
**How:**
1. Add `adversarial_agent_count: int` (0–3, default 0) to the simulation launch config (`POST /api/simulation/start` schema, `<sim_dir>/config.json`). At agent initialization, the last `adversarial_agent_count` agents in the roster receive a modified system prompt injected alongside their standard archetype prompt: "You are designated as an adversarial participant in this simulation. Your role is to challenge the emerging consensus as forcefully and coherently as possible. Identify the weakest arguments supporting the majority position and attack them directly. You may change your own stated stance if the evidence genuinely compels it, but your default orientation is challenge and critique." The adversarial prompt is injected *after* the archetype definition — so the agent retains its archetype persona (Financial Analyst, Risk Manager, etc.) but is explicitly tasked with challenge. Store `adversarial_agents: [agent_name, ...]` in `<sim_dir>/config.json` so the result view knows which agents were adversarial. In each round's post generation, adversarial agents are prompted with the current consensus state explicitly ("Current consensus: YES at 68%") so their challenge is targeted.
2. After simulation completion, compute the **consensus robustness score** in `<sim_dir>/adversarial_report.json`: `{adversarial_count: int, pre_adversarial_consensus: {direction, strength_at_halfway_round}, final_consensus: {direction, strength_at_final_round}, robustness: "high"|"medium"|"low"|"collapse", verdict: str, key_moments: [{round, event: "adversarial_challenge_shifted_consensus"|"adversarial_challenge_absorbed", yes_before, yes_after, agent_name}]}`. Robustness thresholds: `high` = final consensus within 10% of halfway-point consensus; `medium` = 10–25% drift; `low` = >25% drift but same direction; `collapse` = consensus direction reversed. Add `GET /api/simulation/:id/adversarial-report` returning this structure. In the simulation results view, add an "Adversarial" badge on the simulation card (alongside "Clean" / "Flagged") when `adversarial_agent_count > 0`. In the results detail view, show the adversarial summary above the belief drift chart: "Stress-tested with [N] adversarial agent(s) — consensus robustness: HIGH / MEDIUM / LOW / COLLAPSE." Show adversarial agents in the interaction network with a red border on their nodes (distinct from coalition coloring).
3. In Step 2 (Configure), add an "Adversarial mode" toggle below the agent count slider. When enabled, a stepper appears: "Adversarial agents: 1 / 2 / 3" with a tooltip: "These agents will actively challenge the emerging consensus rather than engaging neutrally." Show a note: "Adds approximately [N × 20%] to token usage — adversarial agents generate longer, more argumentative posts." In the gallery explore view, add an "Adversarial" filter chip (alongside "Clean," "Annotated," "Trending") that shows only stress-tested simulations. In the article generator, inject adversarial context as structured input: "Adversarial mode: [N] agents designated as challengers. Robustness verdict: [high/medium/low/collapse]. Key moment: [round R — adversarial challenge [absorbed/shifted] consensus by [X%]]." The generated article addresses robustness directly: "The 71% YES consensus held under direct challenge from two adversarial agents through round 12, suggesting the finding is scenario-driven rather than an artifact of unchallenged groupthink." Document `adversarial_agent_count` and `GET /api/simulation/:id/adversarial-report` in the OpenAPI spec under a new "Adversarial Mode" tag. Add a "Stress-Testing Simulations" section to `docs/FEATURES.md` explaining robustness scores and when to use adversarial mode.

### 3. Weekly Simulation Digest

**Type:** Feature / Community
**Effort:** Small (hours)
**Impact:** The gallery at `/explore` is a full chronological stream. The trending sort (PR #78) surfaces the most-active simulations. But neither provides a curated weekly summary — a "this week in MiroShark" view that a researcher, community member, or new visitor can scan in 60 seconds to understand what the platform has been doing. The Weekly Digest generates this automatically every Monday from gallery data: top 5 simulations from the past 7 days, ranked by a composite engagement score (surface-stats view count + share count + annotation count + coalition complexity). Each simulation gets a one-line summary, its consensus outcome, and its single most interesting metric. The digest is available at `/digest/latest` (always the current week) and `/digest/week/YYYY-WNN` (permanent weekly archives). `GET /api/digest/weekly` returns the same data as JSON, making it consumable by external newsletters, Notion integrations, and the Aeon agent's push-recap skill. For new visitors arriving from search or social, the digest is a faster on-ramp than the full gallery — five examples of what MiroShark actually produces, pre-curated by engagement signal.
**How:**
1. Add `GET /api/digest/weekly?week=YYYY-WNN` (defaulting to the current ISO week). The endpoint scans all published simulations with `published_at` in the 7-day window, computes a composite engagement score per simulation: `score = (surface_stats.watch_count * 3) + (surface_stats.share_count * 5) + (surface_stats.embed_count * 4) + (annotation_count * 2) + (coalition_count * 1)` (all fields gracefully defaulting to 0 if absent). Returns the top 5 by score: `{week: "2026-W20", generated_at: ISO, simulations: [{rank, sim_id, scenario_excerpt (first 120 chars), final_consensus: {direction, strength}, consensus_velocity, key_metric: {name, value, label}, engagement_score, sim_url}]}`. The `key_metric` field auto-selects the most interesting metric for each simulation: whichever of `consensus_velocity`, `polarization_index`, `coalition_count`, or `annotation_count` is most extreme relative to that week's distribution. Cache the response in `<data_dir>/digest_cache/<week>.json` with a 6-hour TTL (recompute when stale). Add `GET /api/digest/archive` returning `{weeks: [{week, sim_count, top_scenario_excerpt, generated_at}]}` — a list of all available digest weeks for archive navigation.
2. Add a `/digest/latest` route in the frontend that renders a curated weekly digest page. Layout: a header section ("Week of May 11–17, 2026 — 5 simulations highlighted from [N] published this week"); below it, 5 simulation cards in rank order. Each card: rank badge (1–5 in a colored circle), scenario text (first 120 chars), consensus outcome badge (YES/NO/NEUTRAL + strength percentage), the key metric callout ("Fastest consensus: round 4"), and a "View simulation →" link. At the bottom of the page, a "Browse all [N] simulations from this week →" link that navigates to `/explore` pre-filtered to the past 7 days. Add a "This Week's Digest →" entry point on the gallery `/explore` page: a pinned banner at the top of the results list (not a sidebar — inline with the feed) showing the #1 ranked simulation from the current week's digest with a "See full digest" link. Add a navigation item in the gallery toolbar: "Digest" alongside "Explore" and "Trending." Add `/digest/latest` and `/digest/week/:weekId` to the Vue Router with `<title>` tag support for SEO (`"Week of May 11–17 — MiroShark Digest"`).
3. In the simulation detail view, when a simulation is ranked in a weekly digest, show a "Featured in Week [N] digest (#[rank])" chip below the scenario title — linking back to the digest page. Add `GET /api/digest/weekly` to the OpenAPI spec under a new "Digest" tag with the full `WeeklyDigestResponse` schema. In the RSS feed (PR #60), add the digest as a meta-feed: `GET /api/feed.atom?digest=true` returns an Atom feed where each entry is a weekly digest summary rather than an individual simulation — one entry per week, published on Monday. For digest-feed subscribers, this provides a weekly newsletter-level summary at the RSS layer without requiring any additional tooling. Document the digest route and RSS variant in `docs/FEATURES.md` under "Weekly Digest." In the sitemap (PR #82), include `/digest/latest` and the last 4 weekly archive URLs as `<loc>` entries with `changefreq: weekly` and `lastmod: <Monday date>`.

### 4. Curator Collections

**Type:** Community / Feature
**Effort:** Small (hours)
**Impact:** The gallery is comprehensive and searchable. Creator Profiles (suggested May 10) organize by authorship. Neither lets a researcher build an *argument* from a curated selection of simulations: "here are the 6 governance simulations that informed this paper," or "here are my best crypto market sims from Q1," or "a beginner's tour of MiroShark outputs, ordered for pedagogical value." Curator Collections fills this gap with the lightest possible implementation: a named, ordered list of published simulation IDs. A collection has a title, optional description (200 chars max), an ordered sim_id list, and a public/private toggle. `/collections/<id>` renders the collection as a curated gallery page — simulation cards in the curator's chosen order, with the collection title and description at the top. The collection URL is shareable and linkable. For academic use, a collection is a citation bundle: "see simulations 1–6 in this collection for all evidence supporting the hypothesis." For community use, it's a curation layer that the algorithmic gallery can't replicate.
**How:**
1. Add a `collections` persistence layer — JSON files in `<data_dir>/collections/<id>/meta.json`: `{id (8-char hex), title (max 60 chars), description (max 200 chars, optional), sim_ids: [ordered list], is_public: bool, created_at: ISO, updated_at: ISO}`. Add REST endpoints: `POST /api/collections` (create — accepts title, description, sim_ids, is_public; validates that all sim_ids are published), `GET /api/collections/:id` (returns meta + sim summaries: scenario_excerpt, final_consensus, model, round_count for each sim_id in order), `PATCH /api/collections/:id` (update title, description, is_public, sim_ids), `DELETE /api/collections/:id`. Add `GET /api/collections` (list all public collections, sorted by updated_at desc, with sim_count and title). No authentication required — collections are first-come, first-served per instance, same as creator handles. Add `GET /api/simulation/:id/collections` returning the list of public collections that include this simulation — enables "also in these collections" cross-linking.
2. Add a `/collections/<id>` route in the frontend. Page layout: a header section (collection title, description, "N simulations · Curated by [handle if set]"); below it, the simulation cards in the curator's specified order (not chronological, not by engagement — strictly the curator's list order). Each card shows rank position as a small number in the top-left corner (1, 2, 3...). Add "Add to collection" as an action on published simulation result pages: a chip button that opens a picker showing the user's existing collections (if any, stored in localStorage for the current instance session) with a checkbox to add/remove the simulation, plus a "+ New collection" option that opens an inline creation form. In the gallery explore view, add a "Collections" tab in the top navigation (alongside "Explore" and "Digest"). The Collections tab shows a list of recent public collections: title, sim_count, first 3 scenario excerpts as a preview. Clicking navigates to `/collections/<id>`. Add a "Share collection" button on the collection page that copies the URL. Add a "Reorder" mode: clicking "Reorder" on the collection management view switches the sim cards to drag-to-reorder mode (CSS `draggable` attribute + a simple `mousedown`/`dragover`/`drop` handler that updates the `sim_ids` array order — no library needed).
3. Add all collection endpoints to the OpenAPI spec under a new "Collections" tag. In `GET /api/simulation/:id` (simulation detail), add a `collection_count: int` field showing how many public collections include this simulation. If `collection_count > 0`, the simulation detail view shows a "Appears in [N] collection(s)" chip that expands to list them. Add `/collections/:id` to the sitemap (PR #82) for public collections: `<loc>/collections/<id></loc>` with `changefreq: weekly` and `lastmod: <updated_at date>`. In `docs/FEATURES.md`, add a "Curator Collections" section: explain what a collection is, how to create and share one, and a note on ordering: "Collections preserve the curator's ordering — they're an argument, not a feed." Include a concrete use case: "A researcher can create a collection of 8 simulations, order them from least to most contested, link the collection URL in a paper, and maintain it as a living reference that updates when they add new simulations." Note that deleting a simulation removes it from all collections it appears in.

### 5. Scenario Pre-flight Analyzer

**Type:** DX
**Effort:** Small (hours)
**Impact:** Every simulation starts with a blank scenario text field and no feedback until the run completes — which takes time and tokens. A first-time user who types "Will AI take everyone's jobs?" and launches a 15-agent simulation discovers at the end that all agents converged to "YES" in round 2 (neutral collapse) because the framing was too one-sided, the stakes too diffuse. The Scenario Pre-flight Analyzer catches this before the run. `GET /api/scenario/analyze?text=<text>` performs lightweight analysis — no LLM call, pure heuristics — and returns: **controversy score** (0.0–1.0, based on lexical polarity balance and hedge-word density), **consensus lean** (YES/NO/NEUTRAL predicted direction with confidence), **recommended agent count range** (minimum for meaningful divergence given the topic breadth), and **quality risk flags** (e.g., "Framing bias: strong [YES] lean detected — consider adding a counterargument or using neutral framing for richer agent disagreement," "Vague scope: no temporal horizon detected — add a deadline for sharper agent positioning"). In Step 1 of the launch wizard, the "Scenario Analysis" panel appears below the textarea (500ms debounce after typing stops) and shows these signals in a compact 2×2 card grid. Users who see "controversy: 0.3 — LOW" and a framing-bias warning can refine their scenario text before launching, reducing first-run abandonment and quality guard failures.
**How:**
1. Add `GET /api/scenario/analyze?text=<text>` as a Flask endpoint. The analysis is pure Python stdlib — no external API call, no LLM. Implement `ScenarioAnalyzer` in `backend/app/services/scenario_analyzer.py` with four components: (a) **Controversy score**: tokenize the text (split on whitespace + punctuation), count positive-sentiment words (a hardcoded list of ~80 terms: "benefit," "opportunity," "growth," "improve," "succeed," "support" etc.) and negative-sentiment words (~80 terms: "risk," "fail," "collapse," "decline," "harm," "lose," etc.). Controversy = `1 - |pos_count - neg_count| / (pos_count + neg_count + 1)`. Range: 0.0 (completely one-sided) to 1.0 (perfectly balanced). (b) **Consensus lean**: `{"direction": "YES"|"NO"|"NEUTRAL", "confidence": "high"|"medium"|"low"}`. Lean = direction of the dominant polarity. Confidence = high if the raw imbalance >70%, medium if 40–70%, low if <40%. (c) **Agent count recommendation**: `{"min": int, "recommended": int}`. Derive from text length and specificity: short, specific scenarios (under 80 words, named entities present) → min 6, recommended 10. Long, vague scenarios (over 150 words, few named entities) → min 10, recommended 15. (d) **Quality risk flags**: a list of `{code: str, severity: "warning"|"info", message: str}`. Flag codes: `framing_bias` (controversy < 0.35), `no_temporal_horizon` (no year/month/date in text), `double_question` (text contains more than one `?`), `too_short` (under 25 words), `too_long` (over 250 words). Return `{controversy_score: float, consensus_lean: {direction, confidence}, agent_count: {min, recommended}, risk_flags: [{code, severity, message}], analyzed_at: ISO}`.
2. In `Step1Scenario.vue`, add a `ScenarioAnalysis.vue` child component that renders below the scenario textarea. The component fires a debounced fetch (500ms) to `GET /api/scenario/analyze?text=<current_text>` whenever the textarea content changes and has at least 15 characters. While fetching, show a subtle "Analyzing..." spinner in the top-right corner of the analysis panel. On result, render a 2×2 card grid: (a) "Controversy" — a horizontal bar (0–100%) colored green (>60%), yellow (35–60%), red (<35%), with the score as a percentage and a one-word label ("Balanced / Moderate / One-sided"); (b) "Consensus lean" — a direction badge (YES green / NO red / NEUTRAL gray) with confidence label; (c) "Recommended agents" — the recommended count as a large number with a "min: [min]" sub-label; (d) "Quality risks" — a count of flags as a number with a color (green = 0, yellow = 1, red = 2+). Below the grid, expand the risk flags as a list: each flag as a one-line warning with an icon (⚠️ warning, ℹ️ info) and the message. Clicking a flag highlights the relevant part of the scenario text (word-level highlight via `<mark>` tags injected into a mirror div). When the analysis panel shows `framing_bias` warning, auto-suggest the `adversarial_agent_count` toggle in Step 2: "Consider enabling Adversarial Mode to counter the detected framing bias." This creates a cross-feature nudge that drives adoption of the adversarial feature.
3. Add `GET /api/scenario/analyze` to the OpenAPI spec under a new "Pre-flight" tag with the `ScenarioAnalysisResponse` schema. Expose all five response fields with full descriptions. Add 15 unit tests in `backend/tests/test_unit_scenario_analyzer.py`: controversy score computation (balanced, one-sided, empty text), lean direction and confidence (high/medium/low thresholds), agent count ranges (short/long/specific text variants), flag detection (each of the 5 flag codes), full response structure validation. In `docs/FEATURES.md`, add a "Scenario Pre-flight Analyzer" section explaining what each signal means and how to use the risk flags to refine a scenario. Include a worked example: original scenario (controversy: 0.2, framing_bias warning), refined scenario after removing one-sided language (controversy: 0.68, no flags). Note explicitly: "The pre-flight analyzer uses heuristic methods and does not make LLM calls — it's instant and free regardless of model settings. Treat it as a rough signal, not a prediction." Add a note: "Low controversy scores predict higher quality-guard pass rates — but a one-sided scenario can still be intentional (e.g., testing how quickly agents resist a weak framing)." Cross-reference the Adversarial Mode documentation for the framing-bias flag.

## Selection Rationale

Five angles distinct from all 25 ideas suggested across May 10, 12, and 14, and from the open chart SVG PR:

- **Private Share Links** (#1) — Publishing is all-or-nothing: gallery-indexed or invisible. There's no middle state for sharing preliminary or sensitive simulations with a specific collaborator without gallery commitment. Token-signed expiring URLs with per-link revocation fill this gap precisely. The chart SVG PR explicitly noted "Private Share Link — not yet built" as the fifth outstanding item from the May-14 batch, confirming this is next.
- **Adversarial Stress-Test Mode** (#2) — The quality guard detects degenerate outcomes. What it doesn't detect is fragile ones: consensus that would have collapsed under direct challenge. Adversarial agents designated at launch test this directly and produce a "consensus robustness" verdict — HIGH, MEDIUM, LOW, or COLLAPSE — that tells researchers whether their finding is scenario-driven or an artifact of unchallenged groupthink. Distinct from Multi-Model Race (isolates model variable), Ensemble Runs (isolates seed variance), and A/B Comparison (compares independent runs).
- **Weekly Simulation Digest** (#3) — The gallery is comprehensive; the trending sort is algorithmic. Neither provides a curated weekly summary that a new visitor can scan in 60 seconds. The digest endpoint + page + RSS variant generates this automatically from engagement signals every Monday. It's the "this week in MiroShark" surface that makes sense for the project at 1,164 stars and growing content volume — the same editorial layer that every platform adds when the feed gets too dense to browse cold.
- **Curator Collections** (#4) — Creator Profiles (May 10) organize by authorship. The trending sort organizes by engagement. Neither lets a researcher build an ordered argument from a curated selection: "here are the 8 governance simulations that informed this paper, ordered from least to most contested." Collections are the citation-bundle primitive — the difference between "link to MiroShark" and "link to these specific simulations in this specific order."
- **Scenario Pre-flight Analyzer** (#5) — The blank scenario field gives no feedback before a run. Quality guard failures and neutral-collapse outcomes are often predictable from the scenario text alone (one-sided framing, vague scope, missing temporal horizon). A pure-heuristic, zero-LLM-cost analyzer that runs in the Step 1 UI catches these before the run starts — reducing first-run abandonment and improving quality guard pass rates. The cross-feature nudge (framing bias → enable Adversarial Mode) also drives adoption of Idea #2.
