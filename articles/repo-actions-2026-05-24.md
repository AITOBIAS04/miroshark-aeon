## Ecosystem Context

MiroShark hits 1,194 stars today. The past week's merges shifted the center of gravity from *distribution surfaces* (badge SVG, BibTeX, WaybackClaw) toward *integrator surfaces*: a machine-readable prediction JSON (#99) shaped for Polymarket connectors, Nemotron-anchored demographic personas (#103) that ground agent archetypes in real census distributions, and a platform aggregate stats API now in review (#105). The pattern is clear — MiroShark is becoming infrastructure that other tools consume, not just a webapp that researchers visit.

That shift surfaces two gaps. First, the consumption layer assumes a developer who can write HTTP calls and parse JSON. An AI assistant user or a non-developer analyst gets no onramp. Second, the production layer assumes every simulation is configured from scratch. There's no vocabulary layer (semantic tags are built but push-blocked), no confidence signal that downstream tools can trust without reading the full report, and no path for new users who face a blank scenario box and don't know where to start.

Today's five ideas address both gaps without touching the already-in-progress work (open PRs #104/#105, and the 23 push-blocked features waiting for GH_GLOBAL). Each is scoped to what an autonomous AI agent can fully implement and commit in one session: clear inputs, deterministic outputs, no design ambiguity, zero new dependencies.

### 1. French (fr) Locale

**Type:** Community
**Effort:** Small (hours)
**Impact:** Issue #95 is an explicit community ask from a willing contributor — accepting it signals that MiroShark welcomes non-English-speaking communities and that the i18n infrastructure (added with the zh-CN toggle in PR #61) is real, not a one-off. French is the second-most-spoken language in European AI research circles and the primary language of several active governance simulation communities (ENA, Sciences Po policy labs). A French locale converts MiroShark from a bilingual (EN/ZH) tool to a trilingual one, opening the French-speaking academic and govtech market that the OriginTrail DKG citation and BibTeX export are already positioned to serve — those users just need the UI to not be in a language they don't use day-to-day. The `fr` locale also serves as a template for future community-contributed locales (DE, ES, PT) because it exercises every translation hook in the codebase.
**How:**
1. Copy `frontend/src/locales/en.js` to `frontend/src/locales/fr.js`. Translate every string key: UI labels (Step 1–3 navigation, button text, input placeholders, error messages, gallery filters, result panel headers, share surface labels, embed dialog text). For simulation-specific vocabulary (archetype names, stance labels, consensus-direction terms), match the canonical French ML/social-science terminology rather than literal translation (e.g. "Bullish" → "Haussier", "Neutral" → "Neutre", "Bearish" → "Baissier", "Archetype" → "Archétype"). Do not translate the project name (MiroShark stays MiroShark), model names, or API endpoints. Add `fr` as a selectable option in the language toggle (`LanguageToggle.vue` or equivalent): three-state flag/chip cycle (EN → ZH → FR → EN). The toggle order should follow geographic time zone (west-to-east: FR before ZH).
2. Update `backend/app/i18n/` (if it exists) or the equivalent backend message dict with French translations for all user-facing error messages returned from the API (validation errors, rate limit messages, publish-gated access errors). These are the strings that appear when a French user gets a 400 or 403 response — leaving them in English breaks the locale consistency. French backend strings follow the same pattern as the Chinese backend strings in PR #62: a `fr` key in each message dict.
3. Add a `docs/LOCALE-fr.md` (or extend `docs/i18n.md` if it exists) documenting: how to switch to French, what's translated and what isn't (API endpoints, model names, archetype IDs are not translated), and how future community contributors can add more locales using `fr.js` as the template. Update `README.md` to add the French flag emoji alongside the existing English/Chinese flags in the language section. Open an issue on the watched repo closing out #95: "French locale landed in PR X — merci!"

### 2. Agent Interaction Graph Export (GraphML)

**Type:** Integration
**Effort:** Small (hours)
**Impact:** The interaction network visualized in `InteractionNetwork.vue` is one of MiroShark's most distinctive outputs — a directed graph where edges represent influence events (agent A's post changed agent B's stance). Today that graph lives only in the browser. A researcher who wants to run centrality analysis (PageRank, betweenness), community detection (Louvain, Girvan-Newman), or temporal graph analysis on the interaction data has to parse `actions.jsonl` manually and reconstruct the graph themselves. GraphML is the lingua franca of graph tools: Gephi opens it directly, NetworkX loads it in one line (`nx.read_graphml(f)`), Cytoscape imports it natively. An endpoint that emits the interaction graph as a GraphML file closes the last-mile gap for network scientists — they click "Download graph" and their analysis environment does the rest. This pairs directly with the BibTeX citation (#96), reproduce.json (#75), and trajectory CSV (#66): a researcher now has the scenario config, the citation, the per-agent time-series, and the interaction graph, all as download-ready files. Zero new dependencies — `xml.etree.ElementTree` ships with Python 3.
**How:**
1. Add `backend/app/services/graphml_export.py` (~150 LoC, pure stdlib using `xml.etree.ElementTree`). The function `build_graphml(sim_id, data_dir)` reads `<sim_dir>/profiles.json` (node list: agent_name, archetype, initial_stance) and `<sim_dir>/actions.jsonl` (edge list: each reply event becomes a directed edge from replying_agent to target_agent, with `round` and `influence_occurred: bool` as edge attributes). Returns a GraphML XML string conforming to the GraphML 1.1 spec: `<graphml>` root with `<key>` declarations for node attributes (archetype: string, initial_stance: string, final_stance: string — read from last round of `trajectory.json`) and edge attributes (round: int, influence: boolean). Nodes also carry `final_stance` and `flip_count` (from trajectory). Edges are directed (`edgedefault="directed"`). If `actions.jsonl` is absent or empty, returns a node-only graph (valid GraphML). Add `GET /api/simulation/:id/network.graphml` to `simulation.py`: calls `build_graphml`, returns a `text/xml` response with `Content-Disposition: attachment; filename="miroshark-<sim_id>-network.graphml"`. Register no new blueprint — add to the existing simulation blueprint.
2. In `Step3Simulation.vue`, add a "Download Graph" button in the share/export toolbar (alongside the existing Trajectory CSV, BibTeX, and Notebook export buttons). Uses the same download-link pattern: `<a :href="'/api/simulation/' + simId + '/network.graphml'" download>`. Add a tooltip: "Download interaction graph in GraphML format — open in Gephi, NetworkX, or Cytoscape." In `EmbedDialog.vue`, add a "GraphML" chip in the data exports section alongside the existing export chips. In `archive_bundle.py` (the Archive Bundle service from PR #92), add `network.graphml` to the bundle manifest: call `build_graphml` and include the result as `network.graphml` inside the ZIP.
3. Add `GET /api/simulation/:id/network.graphml` to the OpenAPI spec under the existing "Exports" or "Data" tag: `produces: text/xml`, response schema description notes the GraphML 1.1 format, key attributes (archetype, initial_stance, final_stance, flip_count for nodes; round, influence for edges), and links to the GraphML 1.1 spec URL. Add 10 unit tests in `backend/tests/test_unit_graphml_export.py`: empty actions returns node-only graph, directed edges present, `<key>` declarations for all 5 attributes, node attribute values (archetype, final_stance), edge attribute values (round, influence), valid XML parse (no ElementTree parse error), missing profiles.json returns empty graph gracefully, missing trajectory.json omits final_stance gracefully, bundle inclusion (graphml file present in archive ZIP), route decorator presence.

### 3. Simulation Confidence Score

**Type:** Feature / Analytics
**Effort:** Small (hours)
**Impact:** MiroShark's current quality signal is `quality_health` (a coarse green/yellow/red label based on consensus stability). Downstream integrators — the quant tools consuming `signal.json` (#91), the governance teams reading the calibration panel (#99), the curators building Collector Collections — need a finer-grained, single-number trust signal: "can I rely on this simulation's consensus, or was it noisy?" The Confidence Score is that signal: a 0–100 integer derived from four observable properties of the completed simulation. It requires no new data collection — everything it needs already exists in `trajectory.json`, `coalitions.json`, and `adversarial_report.json` (if run). A score of 85 means: the consensus stabilized early, agent coalitions were coherent, the adversarial variant didn't collapse it, and a meaningful fraction of agents participated. A score of 23 means: consensus oscillated through the final round, coalitions were fragmented, and adversarial injection caused collapse. Downstream tools can filter by `?min_confidence=70` to build pipelines that only act on high-confidence sims — this is the signal that makes `signal.json` production-safe rather than experimental.
**How:**
1. Add `backend/app/services/confidence_service.py` (~200 LoC, pure stdlib). The function `compute_confidence(sim_id, data_dir)` reads `trajectory.json` and computes four component scores, each 0–25: **Stability** (25 × (1 − σ/range) where σ is the round-over-round std dev of `yes_pct` in the last half of rounds and range is the full yes_pct range; clamped 0–25); **Convergence** (25 × max(0, 1 − |final_yes_pct − 50| / 50) inverted — a sim that ends at 80% YES scores higher than one at 52%; formula: if |final − 50| > 20pp, convergence = 25 × (|final − 50| − 20) / 30, clamped 0–25); **Participation** (25 × min(1, active_agents / total_agents) where `active_agents` = agents with ≥ 2 posts across all rounds); **Robustness** (read `adversarial_report.json` if present: robustness_tier maps to score — "high" = 25, "medium" = 18, "low" = 8, "collapse" = 0; if no adversarial report, use coalition modularity from `coalitions.json` as proxy: modularity × 25, clamped 0–25; if no coalitions either, default = 15). Sum the four components (0–100). Write `{confidence_score: int, components: {stability: int, convergence: int, participation: int, robustness: int}, computed_at: ISO}` to `<sim_dir>/confidence.json`. Add `GET /api/simulation/:id/confidence` returning this object. Cache is the `confidence.json` file itself (recompute if absent, never recompute if present — scores are deterministic post-run).
2. In the gallery card payload (`GET /api/simulation/public`), include `confidence_score: int|null` (null if no `confidence.json`). In `ExploreView.vue`, add a `?min_confidence=N` filter: a slider or preset chips (Any / 60+ / 75+ / 90+) rendered below the existing filter row. In `SimulationCard.vue`, show a confidence chip when `confidence_score ≥ 70`: a small badge `🎯 87` (icon + number) in the card footer alongside the calibration chip and upvote chip. In `Step3Simulation.vue`, add a Confidence panel in the results header bar (alongside the quality_health chip): show the 0–100 score with a color band (green ≥ 75, yellow 50–74, red < 50) and an expandable breakdown showing the four components as a mini horizontal bar chart. In `signal.json` (the Trading Signal JSON endpoint from PR #91), add `confidence_score: int` to the payload so quant tool integrators get it in the same call.
3. In `EmbedDialog.vue`, add `confidence_score` to the embed metadata section. Add `GET /api/simulation/:id/confidence` to the OpenAPI spec under the "Analytics" tag with a `ConfidenceResponse` schema. Add 12 unit tests in `backend/tests/test_unit_confidence.py`: stability score for stable trajectory (low σ), stability score for volatile trajectory (high σ), convergence score for high-consensus (80% YES), convergence score for split outcome (51% YES), participation with all agents active, participation with half agents silent, robustness from adversarial report (all 4 tiers), robustness fallback from modularity, robustness fallback with neither source, total clamped to 0–100, cache read (existing confidence.json returned without recompute), route decorator presence.

### 4. Scenario Template Library

**Type:** Community / DX
**Effort:** Small (hours)
**Impact:** The most common onboarding failure for new MiroShark users is the blank scenario text box: "What do I type here?" The current workaround is to look at published gallery simulations for examples — but that requires finding the gallery, browsing until something looks relevant, and manually copying the scenario structure. A template library removes this friction entirely: operators pick a pre-built scenario starter (Earnings Announcement, Policy Vote, Product Launch, Election Outcome, Market Crash, Regulatory Ruling, Scientific Breakthrough, Technology Launch) and get a complete, parameter-ready scenario text they can edit rather than write from scratch. Templates also standardize the vocabulary across simulations in the same category, which directly improves the tag system and Archetype Leaderboard (apples-to-apples comparisons require consistent scenario framing). For the "Will a MiroShark simulation be cited in a peer-reviewed paper?" hyperstition, a researcher who can generate a governance simulation from a policy-vote template in 30 seconds is categorically more likely to run and cite it than one who has to compose the scenario from scratch. Zero new dependencies — templates are JSON files in the repo.
**How:**
1. Create `backend/templates/` directory with 8 JSON files, one per scenario category: `earnings_announcement.json`, `policy_vote.json`, `product_launch.json`, `election_outcome.json`, `market_crash.json`, `regulatory_ruling.json`, `scientific_breakthrough.json`, `technology_launch.json`. Each file: `{id: str, name: str, category: str, description: str (1–2 sentences), scenario_template: str (the full scenario text with {{PLACEHOLDER}} tokens for operator-specific values — e.g. "{{COMPANY_NAME}} announced Q2 earnings of ${{EPS}} per share, {{BEATS/MISSES}} analyst estimates by {{PCT}}%..."), suggested_agents: int, suggested_rounds: int, suggested_archetypes: [str] (3–5 archetype names that fit this category), tags: [str] (2–3 suggested tags)}`. Scenario texts should be 80–150 words, realistic, and immediately usable after filling in the placeholders. Add `GET /api/templates` returning `{templates: [{id, name, category, description, suggested_agents, suggested_rounds, tags}]}` — omits `scenario_template` from the list view. Add `GET /api/templates/:id` returning the full object including `scenario_template`. No admin gating — templates are read-only public data.
2. In `Step1GraphBuild.vue` (or the equivalent new-simulation form), add a "Browse Templates" button alongside the scenario textarea. It opens a modal with: a category filter row (All / Earnings / Policy / Product / Election / Market / Regulatory / Science / Tech), a grid of template cards (name, category chip, description, tags), and a search input. Clicking a template card populates the scenario textarea with the `scenario_template` text and pre-fills `suggested_agents` and `suggested_rounds` into their respective inputs. A "✏️ Edit this template" UX note appears above the textarea after insertion: "Customize the {{PLACEHOLDERS}} before running." The modal is dismissable (click outside or ✕). In `ExploreView.vue`, add a "Templates" navigation chip alongside Tags, Collections, Digest, pointing to `/templates`.
3. Add `TemplatesView.vue` as a new page at `/templates`: grid of all templates with category filters, matching the modal design (reuse the same component). Each card links to the new-simulation form pre-filled with that template (`/new?template=earnings_announcement`). The router handles the `?template=` param in `Step1GraphBuild.vue` — on mount, if the param is present, fetch `GET /api/templates/:id` and auto-fill. Add `GET /api/templates` and `GET /api/templates/:id` to the OpenAPI spec under a new "Templates" tag with `TemplateListResponse` and `TemplateDetail` schemas. Add `/templates` and a `?template=:id` param to Vue Router. Update `docs/FEATURES.md` with a "Scenario Template Library" section listing all 8 categories and the placeholder convention.

### 5. Embed Referrer Analytics

**Type:** Feature
**Effort:** Small (hours)
**Impact:** MiroShark's embed surface (the `<iframe>` snippet from EmbedDialog) is the highest-reach distribution channel — a single embed on a high-traffic Substack or Notion workspace can generate thousands of impressions with no operator action. The embed analytics surface (`surface_stats.py`) counts total embed hits per simulation, but it tells operators nothing about *where* their simulations are embedded. An operator who embedded a governance simulation and sees "847 embed hits" has no way to know if those came from one viral Substack post, a dozen GitHub READMEs, or a corporate Notion workspace — information that would completely change how they follow up. Embed Referrer Analytics closes this gap: the `/embed` route logs the `Referer` HTTP header domain per request into a per-simulation `embed_referrers.json` file, and a new admin-gated endpoint exposes the aggregated domain table. This is a pure observability add — no change to the embed surface itself, no new dependencies, no impact on embedding sites (Referer is already sent by browsers automatically).
**How:**
1. In the `/embed/<sim_id>` route handler (wherever the embed iframe HTML is served), add Referer header extraction before returning the response: `referer = request.headers.get('Referer', '')`. Parse the domain using `urllib.parse.urlparse(referer).netloc` (stdlib, no new imports). If `netloc` is non-empty and not one of `['', 'localhost', '127.0.0.1']`, update `<sim_dir>/embed_referrers.json` atomically: read existing `{domain: {count: int, first_seen: ISO, last_seen: ISO}}` dict (empty dict if absent), increment `count` for the domain, update `last_seen`, write back. The read-modify-write must be atomic (write to a `.tmp` file, then `os.replace`). Add `GET /api/simulation/:id/embed-analytics` (admin-token gated) returning `{sim_id, total_embed_hits: int (from existing surface_stats), domains: [{domain: str, count: int, first_seen: ISO, last_seen: ISO}], domain_count: int}` sorted by `count` descending. The `total_embed_hits` value is pulled from `surface_stats.json` (existing infrastructure) — no double-counting.
2. In `Step3Simulation.vue`, in the Embed section of the results panel (the section showing the `<iframe>` snippet and EmbedDialog), add a "Referrers" chip/badge when `domain_count > 0`: "Embedded on N domains →" that opens a mini table overlay or drawer showing the domain list with counts and last-seen dates. The overlay is admin-gated in the UI (only visible when the admin token is present in localStorage — same pattern as the existing admin-gated delete and publish buttons). In `EmbedDialog.vue`, add a footer line: "Analytics available after embedding — check the Embed Referrers panel in the sim results page." This hints at the feature without showing raw data in the unauthenticated embed dialog.
3. Add `GET /api/simulation/:id/embed-analytics` to the OpenAPI spec under the "Analytics" tag with an `EmbedAnalyticsResponse` schema: `{sim_id: string, total_embed_hits: integer, domain_count: integer, domains: [{domain: string, count: integer, first_seen: string, last_seen: string}]}`. Add 10 unit tests in `backend/tests/test_unit_embed_analytics.py`: Referer with full URL extracts domain correctly, Referer with subdomain preserved, empty Referer skipped, localhost Referer skipped, 127.0.0.1 Referer skipped, atomic write (no partial-write corruption), count increments on repeat hit from same domain, multiple domains tracked independently, `GET /api/simulation/:id/embed-analytics` response shape, admin-token guard (returns 403 without token). In `docs/FEATURES.md`, add an "Embed Referrer Analytics" section: explains the Referer logging, the admin-gated endpoint, and a note — "Referrer logging is passive and requires no change to embedding sites. Browsers send the Referer header automatically for iframe embeds unless the embedding page sets `Referrer-Policy: no-referrer`."

## Selection Rationale

Five ideas distinct from all ideas suggested May 18–22 and from open PRs #104/#105:

- **French locale** (#1) — Responds directly to issue #95 (explicit community ask). The zh-CN infrastructure (PR #61) already handles multi-locale routing; `fr.js` is additive, zero-risk. First European locale opens French academic and govtech communities who are exactly the audience the BibTeX + DKG citation stack is built for.

- **GraphML export** (#2) — Every prior export (trajectory CSV, notebook, BibTeX, archive bundle) serves a different consumer type. GraphML serves network scientists — people running centrality, community detection, and temporal graph analysis. Gephi and NetworkX are the standard tools; GraphML is their native format. Closes the last-mile gap for graph researchers without requiring them to parse `actions.jsonl`.

- **Confidence Score** (#3) — The quality_health field (green/yellow/red) is too coarse for downstream integrators who need a single number to gate automated workflows. A 0–100 score derived from stability + convergence + participation + robustness gives quant tools and governance teams a trust threshold to filter on. Lands inside `signal.json` at zero additional API cost.

- **Scenario Template Library** (#4) — The blank scenario box is the primary onboarding failure mode. Templates convert "I don't know what to type" into "customize this" — a much lower activation barrier. Standardized scenario framing across templates also makes the Archetype Leaderboard and Tag System more meaningful (comparable runs produce comparable data).

- **Embed Referrer Analytics** (#5) — The embed surface is the highest-reach distribution channel, but operators have zero visibility into where their simulations are actually being embedded. Referer logging is a pure observability add — no impact on embedding sites, no new dependencies — that finally answers "which Substack post is driving my embed traffic?"
