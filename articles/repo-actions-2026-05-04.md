### 1. Private Impact Mode — Relational Network Simulation

**Type:** Feature
**Effort:** Medium (1–2 days)
**Impact:** Every MiroShark simulation today models public opinion: what do people say on Reddit and Twitter? Issue #70 identifies the adjacent use case that MiroShark currently can't touch — what do people *do* privately in response to a decision? A CEO buys a luxury car; a founder raises a round; a manager rolls out a return-to-office policy. The simulation question isn't "what does Twitter say" but "what do employees, clients, and partners actually feel and do?" Private Impact Mode adds a second simulation type alongside the existing public-opinion mode. It unlocks corporate, HR, and institutional use cases that have no current home in MiroShark — and Cyril has already written the backend architecture and is willing to collaborate. The activation cost is unusually low.
**How:**
1. Add a "Simulation Type" selector in the launch config (Step 1): **Public Opinion** (existing mode) vs **Private Impact** (new). In Private Impact mode, the "Context" field is reframed as "Decision under evaluation" and the agent count maps to the size of the affected network. Extend the agent profile model with three relational fields encoded directly into the agent persona: `trust_level` (0–1 trust in the decision-maker), `equity_tolerance` (how much inequity this agent accepts before acting), `institutional_loyalty` (strength of allegiance to the organization vs. personal values). These fields are derived from the scenario text by the profile-generation LLM call — no manual input from the user.
2. Add a new action vocabulary for Private Impact agents — replacing/extending the existing PUBLIC_POST/RETWEET/LIKE set: `CONFRONT` (direct challenge to decision-maker), `COALITION_BUILD` (organizes peers), `SILENT_LEAVE` (quits or withdraws without confrontation), `VOCAL_SUPPORT` (publicly backs the decision), `DO_NOTHING` (passive acceptance). Each round, agents choose an action based on their profile fields, the decision scenario, and visible actions of agents within distance 1 of their relational graph. Distance-2 cascade: agents who observe coalition building or confrontation in their network have their `trust_level` shifted ±0.1 per round based on the dominant action signal they see.
3. Adapt the output layer for Private Impact semantics: belief drift chart relabels axes as "Resistance ↔ Support" instead of "YES ↔ NO price." The generated article uses a private-impact prompt template: "Following [decision], [N] stakeholders in the simulated network produced [action distribution]. The dominant response was [top action] by round [N], driven by [top agents]. [Resistance/Support] peaked at [X]% by the final round." Add a "Relational Network" view tab (beside "Interaction Network") showing agent clusters by trust_level (red = low trust, green = high trust) at the final round. In the OpenAPI spec, document the new simulation_type field and the Private Impact action vocabulary. Reference Cyril's issue in the PR description.

### 2. Simulation Impact Scorecard

**Type:** Feature / DX
**Effort:** Small (hours)
**Impact:** MiroShark produces a generated article and belief drift chart for every simulation. These are good for narrative interpretation but give no quantitative anchor. A researcher who ran 8 simulations over 6 weeks has no way to compare their outputs numerically. Issue #70 identifies this gap — Cyril built MiroResult as a separate tool specifically to score and interpret simulation outputs. Rather than requiring a separate tool, a built-in Impact Scorecard computes four normalized metrics from existing simulation data and surfaces them as a color-coded dashboard tile on the results page. It makes every simulation output machine-comparable and gives non-technical stakeholders a single number to anchor on. Exportable as JSON for programmatic use.
**How:**
1. After simulation completion, compute four metrics from the existing `trajectory.json` and agent data: **(a) Polarization Index** — final Gini coefficient of the YES/NO/NEUTRAL belief distribution (0 = uniform consensus, 1 = maximally polarized). **(b) Influence Concentration** — Herfindahl–Hirschman Index of the top-10 agents' share of total posts in the final 3 rounds (high = narrative dominated by a few agents). **(c) Consensus Velocity** — rounds to first consensus event divided by total rounds (lower = faster convergence). **(d) Narrative Volatility** — average absolute belief-change rate per round across all agents (higher = more turbulent simulation). Normalize each to 0–100. Write a `POST /api/simulation/:id/scorecard` endpoint that computes and caches these four values in `<sim_dir>/scorecard.json`. Call it automatically at simulation completion alongside the article generation step.
2. In the simulation results view, add a "Impact Scorecard" tile in the sidebar (or as a collapsible section below the belief drift chart). Display as a 2×2 grid of metric cards: each card shows the metric name, numeric score (0–100), a color-coded tier label (LOW / MODERATE / HIGH / CRITICAL with red/yellow/green coloring based on fixed thresholds), and a one-line plain-English interpretation: "Consensus was reached quickly and held — low narrative conflict." "A small number of agents dominated the final narrative — concentrated influence." In the gallery explore view, add the top scorecard metric as a chip under each gallery card (e.g., "⚡ High Volatility" or "🟢 Fast Consensus") — turns the gallery into a discovery surface for simulation dynamics, not just topics.
3. Add `GET /api/simulation/:id/scorecard` to the OpenAPI spec returning `{polarization_index, influence_concentration, consensus_velocity, narrative_volatility, computed_at}`. Add a "Download scorecard" button exporting the four metrics as JSON. In the article generator prompt, inject the scorecard as structured context: "Scorecard: Polarization Index 72/100, Influence Concentration 84/100, Consensus Velocity 41/100, Narrative Volatility 58/100." This anchors the article's language — high polarization produces "deeply divided" framing; high influence concentration produces "narrative was shaped by a small cluster of influential agents" framing.

### 3. One-Click Share to X (Tweet Composer)

**Type:** Growth
**Effort:** Small (hours)
**Impact:** PR #71 gave every simulation a shareable URL that pre-fills the launch form. PR #42 gave simulations an OG share card (Open Graph image). The remaining step in the distribution chain is frictionless intent-to-tweet. The MIROSHARK community is active on Twitter — Aaron's own account has the highest-engagement social surface for the project. A "Share to X" button on the simulation results page opens `twitter.com/intent/tweet` pre-populated with a template: `[Scenario excerpt, first 80 chars] → [Consensus outcome, e.g., "73% bearish by round 12"] [share_url] #MiroShark`. The OG card auto-attaches as the link preview (it's already served by PR #42). Zero implementation risk — Twitter intent URLs are pure client-side with no API key, no auth, no secrets. The entire feature is a single anchor tag with a dynamically composed `text=` parameter. For a project where the founder's tweets drive most organic discovery, turning every simulation result into a one-click tweet multiplies the distribution surface by the number of simulations run each day.
**How:**
1. In the simulation results toolbar (alongside "Share", "Fork", "Export"), add a "Post to X" button styled with the X logo. On click, construct a `twitter.com/intent/tweet` URL: `text=` parameter composed as `"[scenario_excerpt_truncated_to_80_chars] → [consensus_summary] [share_url] #MiroShark"`. The `consensus_summary` is derived from the simulation's final belief state: if YES price ≥ 0.65, "→ [YES_pct]% bullish consensus by round [N]"; if NO price ≥ 0.65, "→ [NO_pct]% bearish consensus by round [N]"; otherwise "→ contested — [YES_pct]% YES / [NO_pct]% NO at round [N]". Open the intent URL in a new tab (`window.open(intentUrl, '_blank')`). No API key, no backend changes, no secrets. Client-side only.
2. Add a zh-CN localized version of the tweet template for simulations run in Chinese mode (PR #65). Chinese tweet text: `[场景摘要] → [共识结果] [分享链接] #MiroShark`. The same `window.open` pattern applies — the intent URL text is just in Chinese. Add a tooltip on hover showing the tweet preview text before the user clicks, so they can verify the auto-composed content before posting.
3. In the gallery explore view, add a "Share" icon on each gallery card (visible on hover) that composes the same tweet template but derived from the gallery item's stored metadata (scenario excerpt, final YES price, simulation ID for the share URL). This makes gallery browsing itself a distribution surface — a user who finds an interesting simulation in the gallery can share it to Twitter in one click without having to open the full simulation detail view first.

### 4. Embeddable Simulation Widget (iframe)

**Type:** Integration
**Effort:** Small (hours)
**Impact:** PR #50 added an animated GIF of the belief replay and an in-app Embed dialog for sharing it. But a static GIF isn't embeddable infrastructure — it's a screenshot. A proper embeddable widget is a `<iframe>` endpoint that newsletter authors, blog writers, and Substack publishers can drop into their content with a single snippet. The widget shows the real-time belief drift chart (from the already-computed `trajectory.json`), the final consensus stats (YES/NO price, top agent name, rounds to consensus), and a "Run your own simulation → miroshark.io" CTA link. No auth required — the widget only works for simulations that are already public (published to the gallery). With 1,064 stars and a growing academic cohort, there are already users publishing MiroShark results in blog posts and research writeups who have to screenshot charts. The iframe embed turns their content into live distribution nodes with CTAs back to MiroShark.
**How:**
1. Add a `/embed/<sim_id>` route serving a minimal, frameable HTML page (no MiroShark navbar, no sidebar — just chart + stats + CTA). The embed page: (a) a 400×250px belief drift line chart rendered server-side as an SVG from `trajectory.json` (YES price, NO price, NEUTRAL per round — same data as the main chart but rendered in a stripped-down template); (b) a stats row: final YES price as a percentage + trend arrow, top agent name and archetype, rounds to consensus; (c) a small "Simulate anything →" CTA linking to the MiroShark homepage with a UTM tag (`utm_source=embed&utm_medium=iframe&utm_campaign=<sim_id>`). The route only serves published gallery simulations — returns 404 for unpublished IDs. No login, no JS framework dependency — pure server-side HTML + inline SVG. `X-Frame-Options: ALLOWALL` header to permit embedding.
2. In the simulation results "Share" dialog, add an "Embed" tab (alongside the existing share link, OG card preview, and GIF embed). The Embed tab shows: a live preview of the `/embed/<sim_id>` widget in a bordered iframe preview, and a copy-able HTML snippet: `<iframe src="https://miroshark.io/embed/<sim_id>" width="500" height="300" frameborder="0" allowfullscreen></iframe>`. Include a size preset picker (Small 400×250, Medium 500×300, Large 700×400) that updates the `width` and `height` in the snippet. Add a "Copy snippet" button.
3. Track embed impression analytics by logging a server-side hit counter in `<sim_dir>/embed_stats.json` each time `/embed/<sim_id>` is served. Add `GET /api/simulation/:id/embed-stats` returning `{impressions, first_embedded_at, last_embedded_at}`. In the gallery explore view, show an "Embedded" badge on gallery cards that have been embedded at least once — signals to browsers which simulations have reached external distribution. Document the embed endpoint in the OpenAPI spec and add an "Embed" section to the README.

### 5. Webhook Delivery Log & Replay

**Type:** DX
**Effort:** Small (hours)
**Impact:** The completion webhook (PR #46) fires once when a simulation ends and has no observability layer. A developer building an automation pipeline — n8n workflow, Slack bot, custom API consumer — has no way to debug a failed delivery: did the payload reach their endpoint? What was the HTTP status? Did MiroShark retry? This information is invisible today. Adding a delivery log to the simulation detail view gives webhook consumers exactly what they need to verify and debug integrations: each dispatch attempt shows timestamp, HTTP status code, response body excerpt (first 200 chars), and retry count. A "Replay" button re-fires the original payload to the configured URL — essential for recovering from endpoint downtime without re-running the simulation. This is the DX primitive that turns the completion webhook from a nice-to-have into a trustworthy production dependency. Combined with the mid-run belief threshold alerts (suggested May 2), it creates a complete webhook observability layer.
**How:**
1. In the simulation runner's `_dispatch_webhook` function (from PR #46), wrap the HTTP request in a try/except that captures: `attempt_at` (ISO timestamp), `http_status` (int or None on network error), `response_excerpt` (first 200 chars of response body), `latency_ms`, `error_message` (on exception). Append each attempt to `<sim_dir>/webhook_log.json` as a JSON array of delivery records. On HTTP 5xx or network error, retry once after 30 seconds and log the retry as a second record with `is_retry: true`. Cap at 2 total attempts (initial + 1 retry). Add `GET /api/simulation/:id/webhook-log` returning the full delivery log array.
2. In the simulation detail view, add a "Webhook" section in the sidebar (visible only when a webhook URL was configured for this simulation). The section shows: configured URL (masked: `https://hooks.slack.com/...abc123`), a delivery log table with columns: Attempt, Time, Status (green checkmark for 2xx, red X for error), Latency, and a truncated response preview on hover. If no delivery record exists yet (simulation still running), show "Pending — will fire on completion." Add a "Replay" button that calls `POST /api/simulation/:id/webhook-replay` — this re-sends the stored completion payload (`simulation_complete` event with the same `simulation_id`, `share_url`, and `results` fields) to the configured URL and appends a new delivery record with `is_replay: true`.
3. Add `POST /api/simulation/:id/webhook-replay` to the OpenAPI spec. In the webhook configuration UI (Step 2 – Configure), add a "Test webhook" button (already suggested for mid-run alerts in the May 2 article — this can share the same implementation) that fires a `simulation_test` event payload immediately so the user can verify their endpoint is reachable before launching. Document the full payload schema (`simulation_complete`, `simulation_test`, and the upcoming `belief_threshold_reached` from mid-run alerts) as a unified "Webhook Events" section in the OpenAPI spec and `docs/API.md`.

## Selection Rationale

Five angles distinct from all 15 ideas suggested across May 2 and April 30:

- **Private Impact Mode** (#1) — Adds a second simulation type that unlocks corporate, HR, and institutional use cases completely absent from the current public-opinion model. Issue #70 provides a pre-written architecture and a willing collaborator — the activation cost is unusually low for a feature of this scope.
- **Simulation Impact Scorecard** (#2) — Quantifies what is currently only qualitative. Four normalized metrics (Polarization, Influence Concentration, Consensus Velocity, Narrative Volatility) anchor every simulation in comparable numeric terms — solving the interpretation gap Cyril identified with MiroResult, but built directly into the product rather than requiring a separate tool.
- **One-Click Tweet Composer** (#3) — Zero backend cost, zero API key, maximum distribution leverage. PR #71's shareable links + PR #42's OG card + a `twitter.com/intent/tweet` URL = every simulation result becomes a one-click tweet. For a project where the founder's Twitter account is the primary organic discovery channel, the multiplier effect is immediate.
- **Embeddable Widget** (#4) — The distribution surface that makes external content a referral channel. Newsletter authors, blog writers, and researchers who publish MiroShark results currently have to screenshot charts. A proper iframe embed turns their content into live CTA nodes pointing back to MiroShark — with UTM tracking to measure actual embed-driven traffic.
- **Webhook Delivery Log** (#5) — Closes the observability gap that makes webhooks a toy rather than a dependency. Replay + delivery log + test-before-launch together form the complete DX layer that automation builders need before committing to webhook-based production integrations.
