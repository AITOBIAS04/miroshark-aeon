## Ecosystem Context

MiroShark crosses 1,190 stars today — the BibTeX citation endpoint (PR #96) shipped yesterday, completing the academic-handoff arc that started with reproduce.json (PR #75) and anchored on OriginTrail (PR #84). The share-surface stack is now genuinely deep: 14 publish-gated surfaces, 5 notification channels, a full researcher handoff sequence, on-chain provenance, a machine-readable trading signal, a consensus badge that embeds anywhere a Markdown `<img>` is valid, and a BibTeX entry that drops straight into a `\bibliography{}` block.

What that depth creates is an integration surface problem. MiroShark now produces enough structured output that other tools want to consume it — but consumption still requires a developer who knows the REST API. A quant pulling `signal.json` has to write the HTTP call. A Claude user wants to launch a simulation from a chat window and can't. A curator tagging 30 simulations by topic has to use curator collections as a workaround. A researcher monitoring a recurring scenario has to manually re-run it each week.

Today's five ideas address the access and composability layer: how MiroShark output gets consumed by AI tools, how simulations get semantically labeled, how archetype behavior accumulates into a cross-simulation intelligence layer, how community discussion happens around published simulations, and how a single scenario becomes a persistent time-series signal instead of a point-in-time snapshot. Each is distinct from all 35 ideas suggested across May 10, 12, 14, 16, and 20, and from the open issues and in-progress builds.

### 1. MiroShark as an MCP Tool Server

**Type:** Integration
**Effort:** Small (hours)
**Impact:** The Model Context Protocol (MCP) is the emerging standard for exposing capabilities to AI assistants — Claude, Cursor, Zed, Windsurf, and any client that speaks the MCP JSON-RPC spec can call tools on a running MCP server directly from the conversation interface. A researcher chatting with Claude who wants to run a simulation today has to describe it, copy the scenario text, open MiroShark, paste it, and wait. With an MCP server, they type "run a 20-agent simulation on the US Fed rate decision next week" and MiroShark launches it — the results surface inside the conversation. For developers, an MCP server turns MiroShark into a composable primitive in any AI pipeline: Claude builds the scenario, MiroShark runs it, Claude interprets the output, all in one chain. The MCP surface is the highest-leverage integration gap in the current stack — it puts MiroShark inside every AI interface that already has the user's attention, rather than asking the user to switch contexts. For the "Will a MiroShark simulation be cited in a peer-reviewed paper?" hyperstition, a researcher who runs simulations from their AI assistant is a fundamentally different kind of user than one who navigates to a web form.
**How:**
1. Add `backend/mcp_server.py` using the `mcp` Python SDK (`pip install mcp` — the one new dependency this feature adds; pure stdlib fallback pattern: if import fails, log a warning and exit). Define 5 tools: `launch_simulation(scenario: str, agents: int = 20, rounds: int = 10, model: str = "default")` → calls `POST /api/simulation/start` internally and returns `{sim_id, status, watch_url}`; `get_simulation_status(sim_id: str)` → calls `GET /api/simulation/<id>` and returns `{status, progress_pct, current_round, total_rounds, consensus_direction}`; `get_simulation_results(sim_id: str)` → calls the results detail endpoint and returns `{final_consensus: {yes_pct, no_pct, neutral_pct, direction}, signal: {direction, confidence_pct}, top_influencer: str, quality_health: str, share_url: str}`; `search_gallery(query: str, consensus: str = "", limit: int = 10)` → calls `GET /api/simulation/public` with the filters and returns `[{sim_id, scenario_excerpt, direction, stars_count, published_at}]`; `get_agent_journey(sim_id: str, agent_name: str)` → proxies the agent journey endpoint and returns the full journey object. The server runs on `stdio` transport (the MCP default) — operators launch it as `python mcp_server.py` from the repo root with `MIROSHARK_API_URL=http://localhost:5001` set. Add `MIROSHARK_API_URL` to `.env.example` with a note: "Required for the MCP server — points at your MiroShark instance."
2. Add `docs/MCP.md` with a setup section for three clients: Claude Desktop (paste the `mcpServers` JSON block into `~/.config/claude/claude_desktop_config.json`), Cursor (`.cursor/mcp.json` block), and a generic `mcp-client` test (`npx @modelcontextprotocol/inspector python mcp_server.py`). Include verbatim example exchanges: user asks Claude to "run a 15-agent simulation on the UK election outcome," Claude calls `launch_simulation(...)`, gets back `sim_id`, calls `get_simulation_status(...)` in a loop, then calls `get_simulation_results(...)` and summarizes the output inline. Show the JSON-RPC payload shapes so developers can test outside the MCP clients. Add a `## MCP Server` section to `README.md` feature table with a one-line description: "AI-native access — launch and query simulations directly from Claude, Cursor, and any MCP-compatible assistant." In the EmbedDialog, add an "MCP" integration chip alongside the Webhook and Trading Signal chips: `python mcp_server.py` with a "Copy command" button.
3. Add `backend/tests/test_unit_mcp_server.py` with 12 offline unit tests: the five tool definitions are present in the server's tool list, `launch_simulation` constructs the correct POST payload (scenario, agents, rounds), `get_simulation_results` maps `final_consensus` correctly from the API response shape, `search_gallery` passes `q` + `consensus` as query params, `get_agent_journey` constructs the correct agent-journey URL, all tools handle API errors gracefully (return `{error: str}` rather than raising), `MIROSHARK_API_URL` fallback to `http://localhost:5001`, the server starts and registers all 5 tools without error on a mock transport. Add a `[mcp]` extras group to `pyproject.toml` (or `requirements.txt` annotation): `mcp>=1.0.0` under `# Optional: MCP server support` so it's invisible to operators who don't use MCP but installable with `pip install -r requirements.txt mcp` for those who do. Document `requirements-mcp.txt` as a separate file listing only the MCP dependency so CI can test the core app without it.

### 2. Simulation Tag System

**Type:** Feature
**Effort:** Small (hours)
**Impact:** The gallery has three discovery paths: date (recency), trending (distribution-weighted), and full-text scenario search. None of them support semantic topic grouping. A curator building a collection of governance simulations has to manually identify and select each one. A researcher studying crypto-market consensus has no filter for "crypto" scenarios — they search the scenario text and hope the terminology is consistent. Tags add the semantic layer: at publish time, operators add 1–5 free-text tags ("macro", "governance", "elections", "crypto", "AI regulation") stored in `<sim_dir>/tags.json`. The gallery gains a `?tags=crypto,governance` filter (AND logic). A `/tags` page lists all tags with simulation counts and a trending metric (tag usage growth over the past 30 days). Gallery cards show up to 3 tags as compact chips below the scenario excerpt. For the curator and collector audience — the researchers building citation bundles and Curator Collections — tags are the vocabulary that makes their work reproducible: "I curated this collection of governance simulations" is a claim anyone can verify by filtering `?tags=governance`. For the Aeon agent's weekly digest skill, tags enable a more structured digest: "3 new governance simulations this week, 1 new crypto simulation."
**How:**
1. Add `POST /api/simulation/:id/tags` accepting `{tags: [str]}` (1–5 items, each max 32 chars, alphanumeric + hyphen, lowercased on write). Validates: array length 1–5, no duplicates (after lowercase normalization), individual tag regex `^[a-z0-9][a-z0-9-]{0,31}$`. Writes `{tags: [...], updated_at: ISO}` to `<sim_dir>/tags.json` atomically. Returns `{tags: [...], sim_id}`. Add `GET /api/simulation/:id/tags` returning the same structure (empty array if no tags file). Add `GET /api/tags` returning `{tags: [{tag: str, count: int, recent_count: int (last 30 days), trending: bool}], total: int}` — scans all published sim dirs with tags.json, aggregates counts, marks a tag as `trending: true` if its `recent_count / count` ratio exceeds 0.4 (i.e. ≥40% of its total usage is within the last 30 days). Cache `GET /api/tags` in `<data_dir>/tags_cache.json` with a 1-hour TTL. In `GET /api/simulation/public` (gallery list) and `GET /api/gallery`, add `?tags=tag1,tag2` filter (AND logic — sim must have all listed tags). Add `tags` array to each simulation's gallery card JSON response by reading `<sim_dir>/tags.json` (graceful empty array if absent).
2. In the simulation publish flow (the "Publish to gallery" button in `Step3Simulation.vue`), add a tag input step between the publish confirmation and the final publish call: a tokenized text input (type a tag name + press Enter or comma to add; click ✕ to remove). Shows a live preview of the tag chips. Maximum 5 tags; input disables at limit with a "5 tag max" note. Below the input, show a "Suggested tags" row populated from `GET /api/tags` (top 10 most-used tags as clickable chips — click to add). On mobile, the tag input collapses to a "Add tags (optional) →" expandable row. In `ExploreView.vue`, add a tag filter area below the sort/filter row: a horizontally scrollable row of all active tags from `GET /api/tags`, rendered as filter chips (gray by default, orange when active). Selecting a chip sets `?tags=<tag>` in the URL; selecting multiple chips builds `?tags=tag1,tag2`. On gallery cards (`SimulationCard.vue`), render the first 3 tags as small chips (`text-xs px-1.5 py-0.5 rounded bg-gray-100 text-gray-600`) below the scenario excerpt. A "+N more" label when there are more than 3.
3. Add a `/tags` route in the frontend routing config leading to a new `TagsView.vue` page: a grid of all active tags, each as a card showing the tag name, simulation count, and a "Trending" badge when `trending: true`. Clicking a tag navigates to `/explore?tags=<tag>`. Add "Tags" as a navigation chip on the `/explore` page alongside the existing "Collections" and "Digest" chips. In the OpenAPI spec, add `POST /api/simulation/:id/tags`, `GET /api/simulation/:id/tags`, and `GET /api/tags` under a new "Tags" section. Add `tags: [str]` to the `SimulationCard` response schema. Add 14 unit tests in `backend/tests/test_unit_tags.py`: tag normalization (uppercase → lowercase), regex validation (rejects spaces, special chars, too-long), length validation (rejects 0 and 6 tags), dedup on write, `GET /api/tags` count aggregation, trending-flag threshold, gallery filter AND logic (sim must have all listed tags), gallery filter with missing tags.json (returns sim), route decorator presence, cache write/read/expiry.

### 3. Cross-Simulation Archetype Leaderboard

**Type:** Feature / Analytics
**Effort:** Small (hours)
**Impact:** Every simulation produces per-agent behavior data: influence counts (how many agents changed stance after reading this agent's post), flip counts (how many times this agent's own stance changed), coalition membership, and post counts. This data lives inside individual sim dirs, never aggregated. An operator choosing a 20-agent configuration doesn't know which archetypes have historically been the most influential or the most persuadable — they pick from the preset pool and hope. The Cross-Simulation Archetype Leaderboard aggregates this behavior data across all published simulations, producing a ranked table: "Risk Manager: top influencer in 34% of simulations, average influence rate 2.1× median, flip rate 18% — the most reliably decisive archetype in the corpus." For operators calibrating agent pools, this is empirical guidance replacing guesswork. For researchers, it's the aggregate behavioral signature of each archetype across the full dataset — publishable as a finding in its own right. For the platform, it's a data story: "after 200 simulations, here's what we know about how AI agents of different types actually argue." No external data dependency — purely derived from the sim_dir corpus.
**How:**
1. Add `backend/app/services/archetype_analytics.py` (~250 LoC, pure stdlib). The service scans all published sim dirs, reads `profiles.json` (archetype labels), `trajectory.json` (per-round stance), and `actions.jsonl` (posts + replies) to compute per-simulation archetype statistics: `influence_count` (inbound reply count per agent in each round, summed), `flip_count` (stance transitions), `coalition_member: bool` (present in coalitions.json if it exists), `was_top_influencer: bool` (highest influence_count in the sim). Aggregate across all sims per archetype: `sim_count` (how many sims contained this archetype), `top_influencer_rate` (fraction where this archetype was top influencer), `avg_influence_per_round` (mean influence_count / n_rounds), `flip_rate` (total flips / total rounds in sims containing this archetype), `coalition_rate` (fraction of sims where archetype was in a coalition). Cache the result in `<data_dir>/archetype_analytics_cache.json` with a 6-hour TTL. Add `GET /api/analytics/archetypes` returning `{generated_at: ISO, sim_count_total: int, archetypes: [{archetype: str, sim_count: int, top_influencer_rate: float, avg_influence_per_round: float, flip_rate: float, coalition_rate: float, leaderboard_rank: int}]}` sorted by `top_influencer_rate` descending.
2. Add `ArchetypeAnalyticsView.vue` as a new page at `/analytics/archetypes`. Layout: a page header ("Archetype Intelligence — aggregated across N simulations"), a stat summary row (total sims analyzed, date range covered, most influential archetype chip, most persuadable archetype chip). Below: a sortable table with columns — Rank, Archetype, Simulations, Top Influencer Rate (with a mini bar), Avg Influence/Round, Flip Rate (stance changes), Coalition Rate. Each column header is clickable for sort (asc/desc). Clicking a row opens a detail drawer: "In 47 simulations containing a Risk Manager archetype — top influencer in 16 (34%), flipped stance 28 times across 510 rounds (flip rate: 5.5%), appeared in coalitions 22 times (47%). Most common coalition partner: Financial Analyst (38 co-appearances)." Add a "Cross-Simulation Analytics" chip on the `/explore` page toolbar (alongside Tags, Collections, Digest). Add `/analytics/archetypes` to the Vue Router config. In `frontend/src/api/simulation.js`, add `getArchetypeAnalytics()` API helper.
3. Add `GET /api/analytics/archetypes` to the OpenAPI spec under a new "Analytics" tag with the full `ArchetypeAnalyticsResponse` schema. In the simulation launch form (Step 1 / Step 2), add a "How archetypes perform" collapsible hint beneath the archetype selection: shows the top 3 archetypes by `top_influencer_rate` from the leaderboard as a mini inline table ("Risk Manager: top influencer in 34% of sims"). Fetched from `GET /api/analytics/archetypes` with a 6-hour cache — renders as a gentle recommendation, not a prescriptive override. Add 10 unit tests in `backend/tests/test_unit_archetype_analytics.py`: empty corpus returns empty list, single sim with one archetype produces correct rates, top_influencer_rate computation across 3 sims, flip_rate derivation from trajectory data, coalition_rate when coalitions.json absent vs present, leaderboard_rank ordering, 6-hour cache TTL read/write, route decorator presence, graceful degradation on malformed profiles.json.

### 4. Discussion Threads on Watch Pages

**Type:** Community
**Effort:** Small (hours)
**Impact:** The watch page at `/watch/<id>` is where published simulations get consumed: the belief drift chart, the interaction network, the agent posts, the share buttons. It's currently read-only for visitors — operators annotate (per-round notes), curators collect, community members upvote (May 20 idea #3, not yet shipped). But none of these are conversation. A researcher who finishes reading a 15-round governance simulation and thinks "the Risk Manager's flip in round 8 is the key moment" has nowhere to say so on the page. They tweet it, or they forget it. Discussion threads add a lightweight comments section below the simulation results — no account required, just a display name and comment text. Comments can reference a specific round (`round_reference: 7` renders as a clickable chip "↩ Round 7" that scrolls to that round in the replay player). Operators can delete comments via admin token. Comment counts appear on gallery cards. For the community layer, this is the first surface where visitors can add structured interpretation to a simulation's record — not a vote signal like upvotes, not an operator annotation, but a public conversation that accumulates value over time. For the academic-citation trajectory, a simulation with 12 comments is a different signal of engagement quality than one with zero.
**How:**
1. Add `backend/app/services/comments_service.py` (~200 LoC, pure stdlib). Stores comments in `<sim_dir>/comments.jsonl` — one JSON object per line: `{id: uuid4 hex, author: str (max 50 chars, stripped), text: str (max 500 chars, stripped), round_reference: int|null, created_at: ISO, deleted: bool (default false)}`. Add `POST /api/simulation/:id/comments` accepting `{author: str, text: str, round_reference: int|null}`: validate author (1–50 chars, strip HTML tags via `re.sub(r'<[^>]+>', '', s)`), validate text (1–500 chars, same strip), validate round_reference (null or positive int ≤ n_rounds). Return `{id, author, text, round_reference, created_at}` with HTTP 201. Rate limit: max 5 comments per IP per simulation per hour (track in `<sim_dir>/comment_rate_limits.json`: `{ip_hash: {hour_bucket: ISO date-hour, count: int}}`). Add `GET /api/simulation/:id/comments` returning `{comments: [...], total: int}` — all non-deleted comments sorted by `created_at` ascending. Add `DELETE /api/simulation/:id/comments/:comment_id` (admin-token gated): sets `deleted: true` on the matching line (rewrite the file atomically). In `GET /api/simulation/:id` (detail endpoint), add `comment_count: int` (count of non-deleted entries) and `GET /api/simulation/public` (gallery list), add `comment_count: int` to each card object.
2. In `Step3Simulation.vue` (the simulation results page), add a "Discussion" section below the share/embed panel. The section header: "Discussion ([N] comments)" — `N` from `comment_count`. Renders the full comments list: each comment as a card with author name (bold), `created_at` formatted as "May 22, 2026", round-reference chip ("↩ Round 7" — clicking it emits a `scroll-to-round` event that the replay player listens to), comment text (pre-wrap, no Markdown rendering), and a "Report" link (opens a mailto with the comment ID pre-filled — lightweight, no backend required for basic moderation). Below the list, a comment form: "Name" text input (max 50 chars) + "Comment" textarea (max 500 chars, live char counter in bottom-right) + "Round (optional)" number input (placeholder "Which round are you commenting on?") + "Post comment" submit button. On submit: calls `POST /api/simulation/:id/comments`, appends the new comment to the list without full reload (optimistic UI with rollback on error). Name is persisted in `localStorage` as `commenter_name` so returning visitors don't retype it. In `SimulationCard.vue` (gallery card), add a small "💬 N" comment count chip in the card footer alongside the upvote chip (when `comment_count > 0`).
3. Add `POST /api/simulation/:id/comments`, `GET /api/simulation/:id/comments`, and `DELETE /api/simulation/:id/comments/:comment_id` to the OpenAPI spec under a new "Community" tag with a `Comment` schema: `{id: string, author: string, text: string, round_reference: integer|null, created_at: string}`. Add 14 unit tests in `backend/tests/test_unit_comments.py`: write + read roundtrip, author HTML-strip, text HTML-strip, length validation (author 0/51 chars, text 0/501 chars), round_reference null vs valid vs out-of-range, rate limit (6th comment same IP-hash same hour returns HTTP 429), deleted comments excluded from GET, admin delete sets deleted flag, comment_count in gallery card, route decorator presence. In `docs/FEATURES.md`, add a "Discussion Threads" section explaining the per-simulation comment store, the HTML stripping, the rate limit, and the admin delete path. Note: "Comments are stored in `<sim_dir>/comments.jsonl` — they are included in the Archive Bundle download (PR #92) automatically when the file exists."

### 5. Recurring Simulation Series

**Type:** Feature
**Effort:** Medium (1–2 days)
**Impact:** Every simulation in MiroShark is currently a point-in-time event: launch, run, publish, done. A macro analyst who wants to know "how is AI agent consensus on Fed rate decisions evolving week over week?" has to manually re-launch the same scenario configuration every Monday, then manually compare the results. The Recurring Simulation Series changes this: at launch, operators can set a `recurrence: {interval: "weekly"|"biweekly"|"monthly", end_date: ISO|null}` on any simulation. A prefetch script checks for due recurrences daily and writes trigger files; the simulation runner processes them as automatic re-runs with the same configuration, grouping all runs under a shared `series_id`. The series page at `/series/<id>` shows all runs as a time-series chart: consensus direction + confidence per run date, plus a delta line ("this week: +7pp toward bullish vs last week"). Operators get a "Track this scenario" button on any simulation result page that enrolls it in a series with one click. For practitioners who use MiroShark as a recurring opinion monitor — "what do AI agents think about X this week?" — recurring series turns it into a dashboard product rather than a research tool. For the MIROSHARK token narrative, a publicly visible "weekly macro tracker" with 20 published runs is a different credibility signal than a gallery of isolated simulations.
**How:**
1. Add `recurrence: {interval: "weekly"|"biweekly"|"monthly", next_run: ISO, end_date: ISO|null, series_id: uuid4 hex}` (all optional) to the simulation launch schema (`POST /api/simulation/start`) and `<sim_dir>/config.json`. At launch, if `recurrence` is provided: generate a `series_id` (uuid4 hex), compute `next_run` as `launched_at + interval`, write a `<data_dir>/series/<series_id>/index.json` with `{series_id, interval, scenario_text (first 200 chars), archetype_config, n_agents, n_rounds, model, created_at: ISO, run_ids: [sim_id]}`. Create `scripts/prefetch-recurrence-check.sh`: scans `<data_dir>/series/*/index.json` for entries where `next_run ≤ today` and `(end_date is null or end_date ≥ today)`, and writes `{series_id, next_run, config: <original config object>}` to `.pending-recurrence/<series_id>-<YYYYMMDD>.json`. After Claude finishes, create `scripts/postprocess-recurrence-trigger.sh`: reads each `.pending-recurrence/*.json`, calls `POST /api/simulation/start` with the stored config plus `{series_id: <id>, parent_run_id: <previous_sim_id>}`, appends the new sim_id to `<data_dir>/series/<series_id>/index.json`, updates `next_run` to `now + interval`, and cleans up the `.pending-recurrence/` file. Add `GET /api/series/:series_id` returning the full index: `{series_id, interval, scenario_excerpt, run_count, runs: [{sim_id, run_date: ISO, final_consensus: {direction, yes_pct}, delta_pp: float|null (vs previous run)}], trend: "toward_bullish"|"toward_bearish"|"stable"|"volatile"}`. The `trend` field: compute linear regression slope on `yes_pct` across all runs — positive slope > 0.5pp/run = toward_bullish, etc.
2. Add `SeriesView.vue` as a new page at `/series/:series_id`. Layout: header with scenario excerpt, interval chip (Weekly / Biweekly / Monthly), run count. Main chart: a time-series SVG (same `xml.etree.ElementTree` stdlib approach as `chart_svg.py`) — x-axis is run date, y-axis is `yes_pct` (0–100%), with a shaded consensus-direction band (green above 60%, red below 40%, gray between). Each run is a dot on the line; clicking a dot navigates to `/watch/<sim_id>`. Below the chart, a run history table: date, direction chip (Bullish/Neutral/Bearish), YES%, delta vs prior run ("+7pp ▲" in green, "-3pp ▼" in red), and a "View →" link. The table is sorted newest-first. In `Step3Simulation.vue`, add a "Track this scenario" button (recurring icon + label) below the publish button. It opens a modal with: interval selector (Weekly / Biweekly / Monthly), optional end date picker, and a "Start tracking" confirm button that calls `POST /api/simulation/start` with the current simulation's config plus the recurrence block. Add `/series/:id` to the Vue Router config. In the gallery card and watch page, when a simulation has a `series_id`, show a "Part of series" chip linking to `/series/<id>`.
3. Add `GET /api/series/:series_id` to the OpenAPI spec under a new "Series" tag with `SeriesResponse` and `SeriesRun` schemas. Add `series_id: string|null` to the simulation detail response so the watch page can show the series chip. Add `GET /api/series` (admin-gated) listing all active series: `{series: [{series_id, interval, scenario_excerpt, run_count, next_run, is_active: bool}]}` — for the operator to audit scheduled recurrences. In `docs/FEATURES.md`, add a "Recurring Simulation Series" section: explains the interval options, the prefetch/postprocess script pattern, the series page, and the trend computation. Include a warning: "Series runs consume tokens on each recurrence — set an `end_date` or deactivate a series from the admin panel to avoid runaway costs." Add 15 unit tests in `backend/tests/test_unit_series.py`: `series_id` generation, `next_run` computation for each interval (weekly = +7d, biweekly = +14d, monthly = +1 calendar month), `delta_pp` derivation between runs, `trend` classification (toward_bullish, stable, volatile), `GET /api/series/:id` response shape, run appending on new trigger, end_date check (run not triggered when past end_date), graceful empty series (0 runs), route decorator presence, admin-gated `GET /api/series` requires admin token.

## Selection Rationale

Five angles distinct from all 35 ideas across May 10, 12, 14, 16, and 20:

- **MiroShark as MCP Tool Server** (#1) — Every other distribution idea (badge SVG, Farcaster Frame, Archive Bundle) pushes simulation *output* to readers. The MCP server pushes simulation *execution* to AI assistants. A researcher in a Claude conversation can launch and retrieve results without leaving the chat. This is the access-pattern inversion: instead of MiroShark pulling users to a web UI, MiroShark shows up in whatever AI context the user already inhabits. Distinct from the webhook (outbound trigger from MiroShark), the Trading Signal JSON (machine-readable output), and the launch webhook (HTTP-in trigger) — those all require a developer who knows the API. MCP targets AI agent chains and non-developer users of AI assistants.

- **Simulation Tag System** (#2) — Curator Collections (May 20 skill) groups simulations into named ordered lists for citation bundles. Community Upvotes (May 20 idea #3, not yet built) adds a quality vote per simulation. Tags are the author-defined semantic vocabulary that makes the gallery semantically navigable: filter by "governance" or "macro" or "elections" as a defined corpus rather than a free-text search against scenario text. Distinct because tags travel with the simulation (set at publish time by the operator), appear on cards, and are aggregatable into a `/tags` browse surface — not a collection built by a curator, not a vote signal, but a first-class label that the operator owns.

- **Cross-Simulation Archetype Leaderboard** (#3) — The Agent Heatmap (per-agent stance per round, within one sim), the Coalition Detection (grouping agents within one sim), and the Agent Journey View (one agent's arc within one sim) all operate at simulation scope. The Archetype Leaderboard operates at corpus scope: across every published simulation, which archetypes win? Which flip most? Which form coalitions? This is the first feature that makes MiroShark's aggregate dataset into a publishable finding — not just "run a simulation," but "here is what 200 simulations reveal about how AI agents of each type argue." Distinct from every prior idea because it requires no new user action: it's purely derived from existing sim dirs, accumulated passively as the gallery grows.

- **Discussion Threads on Watch Pages** (#4) — Community Upvotes (May 20 idea #3) adds a single-tap vote signal. Per-Round Annotations (already built, operator-only) add structured operator notes. Curator Collections let curators group and contextualize simulations. None of them are conversation. A discussion thread is the surface where visitors can add interpretive commentary — "the Risk Manager's round-7 flip is the key insight here" — that accumulates as a public record alongside the simulation data. The round-reference field links comments directly to replay player moments, creating a connected annotation layer that any visitor can contribute to. Distinct from annotations (operator-only, round-native JSON structures) and upvotes (single-bit quality signal).

- **Recurring Simulation Series** (#5) — Every prior feature assumes a simulation is a one-time run. The Inbound Launch Webhook (already built) lets external systems trigger new simulations via HTTP — it's still one-shot. Recurring Series is the first feature that transforms MiroShark from a research artifact tool into a continuous signal product: the same scenario re-runs on a schedule, and the results stack as a time-series that shows how AI agent consensus is evolving over time. A weekly "Fed rate decision" tracker with 20 runs is fundamentally different from 20 isolated simulations — it has a narrative arc, a trend line, and a delta-per-week metric. This is the feature that makes MiroShark useful as a persistent market-sentiment dashboard rather than a single-use research tool.
